## 3. What was harder — challenges & lessons

- **A migration's `--force` escape hatch is a footgun without get-or-create.** The R12 HIGH: `Migration::createUitdaging` blind-inserted, so `--force` (advertised as "re-run / correction") duplicated every round and orphaned the prior terms. Fixed to get-or-create via a source-category marker. **Lesson:** a once-off migration's idempotency must live in the *write* (get-or-create on a stable key), not only in a completion flag — the flag protects the default path, the key protects every re-run.
- **A "safe by default" migration default was actually dangerous.** `legacyCategories()` defaulted to *every* `category`, which would convert ordinary blog categories into published uitdagings. **Lesson:** for a destructive once-off, the default must be the *empty/no-op* set requiring explicit opt-in, not the broad set requiring explicit opt-out.
- **Request-rendered blocks need bounded queries.** The single-page entries list shipped `posts_per_page => -1` (fine for the once-off migration's `numberposts => -1`, but not for a per-view block). **Lesson:** `-1` is acceptable in a CLI/once-off, never on a request path — bound request-rendered lists even when "a round won't have that many."
- **Trusted-writer stores still want a read-side defensive guard.** `Placements` trusts 12A.3 ingestion for rank uniqueness, but a dirty duplicate rank-1 would surface two "algehele wenners" and (via 5.8) over-award. The store's authoritative dedup is 12A.3's job, but `arrange()` now defensively collapses to one-per-rank (lowest id) so the display can never lie. **Lesson:** when the authoritative invariant lives in a not-yet-built consumer/writer, add a cheap read-side guard now and document where the real enforcement belongs.
- **Incrementally growing one module across 8 stories means touching earlier stories' tests/code.** The shared `Deadline` extraction edited 12.1's `SinglePage` during 12.2; `Module::register()` grew each story; `Terms` accreted keys. Normal for a module built feature-by-feature — kept additive (new collaborators, optional params) to minimise churn. Same lesson as Epic 11, held.
- **The "build + unit-verified, integration-pending" pattern continues.** The round `tax_query` entry lists, the cap-count query, the entry-time snapshot against a real `Tiers\Api`, the placement reads, and the WP-CLI migration are all unit-tested against mocks — real-WP behaviour is owed to Story 18.8 (now with a growing Challenges tail).
