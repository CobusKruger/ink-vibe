# Story 4.6: Store-UI suppression

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a site owner,
I want WooCommerce storefront UI hidden,
so that the site reads as a community, not a shop. (FR-10)

## Acceptance Criteria

1. **The general WooCommerce storefront surfaces are suppressed so the site reads as a community, not a shop — but ONLY when WooCommerce is active; when WooCommerce is absent the seam is a complete no-op (no hooks, no fatal).** Given WooCommerce active, when a visitor browses, then the shop/catalog page and product archives, the cart page, and the WooCommerce "add to cart" affordances on generic listings are suppressed (redirected away / made non-public), and the WooCommerce account-menu items that do not apply to a community membership site are hidden. Given WooCommerce inactive/absent, when the module registers, then `StorefrontSuppression::register()` wires **zero** hooks and nothing fatals (the `class_exists( 'WooCommerce' )` guard). _[Source: epics.md#Story-4.6 AC ("Given WooCommerce active, when a visitor browses, then cart/catalog/checkout are suppressed beyond the lidmaatskap purchase flow"); project-context.md "Hook, don't edit" (integrate via WooCommerce hooks/filters, never modify plugin files), platform-plugins list (WooCommerce is a commodity plugin — suppress its storefront, don't reimplement); architecture.md AD-5 ("platform store where one exists"); Entitlement `MembershipPlans`/`PurchaseActivation` `function_exists`-guarded WC-availability precedent]_

2. **THE CARVE-OUT (the crux): the lidmaatskap purchase / renewal flow keeps working end-to-end — the checkout page, the lidmaatskap products' add-to-cart, and the PayFast return path are NEVER suppressed.** Given a visitor following the Story 4.2/4.4/4.5 purchase flow (`Ink\Entitlement\PurchaseActivation::purchaseUrl()` → `wc_get_checkout_url()` + `?add-to-cart=<lidmaatskap product_id>` → off-site PayFast → return), when each suppression hook fires, then it exempts: (a) the **checkout page** (the purchase endpoint stays reachable — only the shop/catalog/cart are suppressed); (b) an **add-to-cart request for a configured lidmaatskap product** (the `ink_membership_plan_products` map's product ids, resolved through `MembershipPlans` — the SAME carve-out distinguisher `SubmissionGate::inkProductIds()` already uses); (c) the cart/add-to-cart machinery WooCommerce itself needs to carry a lidmaatskap product to checkout. The distinction is documented explicitly: **shop/catalog/cart pages = suppressed; checkout + lidmaatskap-product add-to-cart = allowed**, identified by lidmaatskap product id (from the 4.1 registry) and by the WooCommerce checkout endpoint. _[Source: epics.md#Story-4.6 ("suppressed BEYOND the lidmaatskap purchase flow"); PurchaseActivation::purchaseUrl() (the live flow: `wc_get_checkout_url()` + `?add-to-cart=<product_id>`); SubmissionGate::inkProductIds()/isInkMembership() (the existing "is this an INK lidmaatskap product?" resolution via `MembershipPlans`); MembershipPlans::OPTION_PRODUCTS (`ink_membership_plan_products` term→product map); project-context.md "Store-UI suppression must NOT break checkout for the lidmaatskap products"]_

3. **Suppression uses documented WooCommerce hooks/filters only — WooCommerce files are never edited and its internal structure is never assumed; each hook + what it suppresses is documented.** Given the "Hook, don't edit" rule, when suppression is implemented, then each surface is closed via a documented WooCommerce (or WordPress-core) hook/filter — e.g. `template_redirect` to redirect the shop & cart pages away (with the checkout + lidmaatskap-product carve-out), `woocommerce_is_purchasable` / catalog-visibility to hide generic add-to-cart, the `woocommerce_account_menu_items` filter to prune inapplicable My-Account tabs, and dequeuing/removing WooCommerce storefront notices where appropriate — and the class docblock + this story's task list name **each hook and exactly what it suppresses**. No `wc_get_product`/page internals are assumed beyond their documented getters; every WC call is behind a guard so a partial/absent WooCommerce never fatals. _[Source: project-context.md "Hook, don't edit … never modify plugin files or assume internal structure", "Pick the correct, documented WooCommerce hooks and DOCUMENT each one and what it suppresses"; Engagement\Comments precedent (a suppression layer: forces flags closed at `PHP_INT_MAX`, removes support, hides chrome — all via documented hooks, touches no plugin files); architecture.md AD-1 (use WP/WC primitives directly inside the module, behind the facade)]_

4. **This is `ink-core` WooCommerce-integration business logic placed in the `Ink\Entitlement` module (the WC seam owner) as a new `StorefrontSuppression` collaborator wired via the existing `Module::register()` — no new module, no theme logic, conflation-clean.** Given the three-layer rule and AD-1, when this story is built, then: (a) the suppression logic lives in a **new `Ink\Entitlement\StorefrontSuppression` collaborator** registered inside the existing `Entitlement\Module::register()` (added next to `PurchaseActivation` — **no** new `addModule` in `ink-core.php`, **no** parallel `add_action` in the bootstrap, **no** logic in the theme / `functions.php`); (b) it touches **zero `Ink\Tiers`** — store suppression is unrelated to writer Gradering (THE conflation rule; Deptrac `Entitlement ⟂ Tiers` stays green); (c) every new `.php` is `<?php` + one `declare(strict_types=1)` + `namespace Ink\Entitlement;` + `defined('ABSPATH')||exit;`, classes PascalCase / methods camelCase, hooks/options `ink_`-prefixed (or WC-own) single-source `public const`s, escape-on-output / sanitise-on-input, no raw `$_POST`/`$_GET`. 5. **Suppression is removal, not new copy — but any user-facing string this introduces (e.g. a redirect notice) is curated Afrikaans (Gate D); where suppression is a bare redirect, it adds NO copy.** Given Quality Gate D, when this story surfaces any string, then it is Afrikaans-source from the terminology registry / `ui-copy-translations.md`, sentence case, zero English leakage, never AI-invented. If a redirect notice is genuinely warranted, register it Afrikaans-source via the existing Notifications/registry pattern with a `[NEEDS HUMAN AFRIKAANS]` placeholder; do not invent it. _[Source: project-context.md Gate D (front end entirely Afrikaans, sentence case, no AI Afrikaans), "Most suppression is removal (no new copy) — keep it that way where possible"; AD-10 terminology registry (never inline a glossary label as a bare literal); ui-copy-translations.md `[NEEDS HUMAN AFRIKAANS]` precedent]_

### Completion Notes

**Architecture decision — placement & carve-out.** Storefront suppression is WooCommerce-integration business logic, so it lives in `ink-core`, not the theme (three-layer rule). It was placed in the `Ink\Entitlement` module — the existing WooCommerce/WC-Memberships seam owner (AD-1/AD-5: "platform store where one exists") — as a **new `Ink\Entitlement\StorefrontSuppression` collaborator**, wired via the existing `Entitlement\Module::register()` next to `( new PurchaseActivation() )->register();`. **No new module, no `addModule` in `ink-core.php`, no bootstrap edit.** Rationale: the carve-out distinguisher ("is this a lidmaatskap product?") is the SAME `MembershipPlans` product-id set `SubmissionGate::inkProductIds()` already uses — co-locating in Entitlement keeps one source of truth and avoids a new module for a thin collaborator. Deptrac stays within `Entitlement → {Kernel, Notifications}`; no new edge, `Entitlement ⟂ Tiers` permanent prohibition untouched.

**WooCommerce hooks used and what each suppresses (AC-3):**
- `template_redirect` (`onTemplateRedirect`) — redirects the SHOP page (`is_shop()`), PRODUCT archives/singles (`is_product()` / `is_post_type_archive('product')`), and the CART page (`is_cart()`) to `home_url('/')`. The general "this is a shop" surfaces.
- `woocommerce_is_purchasable` (`filterIsPurchasable`) — forces any NON-lidmaatskap product un-purchasable (inert generic add-to-cart); lidmaatskap products keep WooCommerce's decision.
- `woocommerce_loop_add_to_cart_link` (`filterLoopAddToCartLink`) — strips the catalog-loop add-to-cart link for non-lidmaatskap products (returns ''); a removal, no new copy.
- `woocommerce_account_menu_items` (`filterAccountMenuItems`) — prunes the inapplicable My-Account storefront tab(s) (`downloads`); keeps orders/payment-methods/account.

**The carve-out (the crux, AC-2) — HOW the allowed membership flow is distinguished from the suppressed general store:** by **two signals**: (1) the **checkout endpoint** — `is_checkout()` is never redirected; only shop/catalog/cart are; (2) the **lidmaatskap product id** — an `?add-to-cart=<id>` request whose id is in the configured `ink_membership_plan_products` set (resolved through `MembershipPlans`, the same set `SubmissionGate` uses) passes through `template_redirect` untouched, and `woocommerce_is_purchasable` / the loop link leave lidmaatskap products intact. So the Story-4.2 flow (`PurchaseActivation::purchaseUrl()` → `wc_get_checkout_url()` + `?add-to-cart=<lidmaatskap product_id>` → off-site PayFast → return) is fully exempt. The `FILTER_SUPPRESS_EXCEPTION` re-open seam (default suppress) mirrors `Comments::FILTER_OPEN_EXCEPTION` for future narrower exemptions.

**WooCommerce-absent = total no-op (AC-1):** `register()` returns early wiring ZERO hooks when `class_exists('WooCommerce')` is false; every handler also `function_exists()`-guards each WC conditional so a partial/loading WC never fatals. No WooCommerce file is edited and no undocumented internal is assumed (only documented hook names + `get_id()` getter).

**Afrikaans/Gate D (AC-5):** suppression is pure REMOVAL — the only front-end effect is a bare `wp_safe_redirect()` to home. **No new user-facing string was introduced**, so there is no Gate-D copy to author and **no `[NEEDS HUMAN AFRIKAANS]` flag is needed** for this story. The `FILTER_SUPPRESS_EXCEPTION` seam is reserved if a future story wants a notice.

**Conflation rule (AC-4):** zero `Ink\Tiers` reference, no `ink_writer_tier` read/write — asserted by a comment-stripped source scan. Deptrac green for `Entitlement ⟂ Tiers`.

**Security:** the add-to-cart product id is read via `filter_input(INPUT_GET/INPUT_POST, …, FILTER_SANITIZE_NUMBER_INT)` coerced to int — **no raw `$_GET`/`$_POST`/`$_REQUEST`** (asserted by a source scan). Redirect uses `wp_safe_redirect` (local-only).
