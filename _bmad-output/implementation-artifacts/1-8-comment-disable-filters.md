# Story 1.8: Comment-disable filters

Status: in-progress

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a site owner,
I want WP comments disabled site-wide in `ink-core` (not the theme) — `comments_open`/`pings_open` forced false for every post type and query, comment support removed from all post types, existing open threads closed, and the comment admin/UI surface hidden — with a single documented programmatic seam that leaves `wp_insert_comment` writes for the sanctioned custom comment types (`ink_reaksie`, `ink_moderator_terugvoer`) untouched,
so that engagement flows only through the structured paths (Gemeenskapsreaksies + the moderator-feedback type), the free-form WP comment UX never appears, and `ink-core`'s own structured-response storage (which reuses the comments table programmatically, not the public form) keeps working.

## Acceptance Criteria

1. **`comments_open` and `pings_open` resolve to false for every post type and query** — Given the comment-disable filters registered by `ink-core`, when any post type is rendered or queried (front end, admin, REST, feeds), then the `comments_open` filter returns `false` and the `pings_open` filter returns `false` regardless of the per-post stored value or post type. The callbacks are `ink_`-prefixed, attached at a late priority so they win over post defaults, and take precedence over the `wp_insert_comment` programmatic path (which does not consult `comments_open` — see AC-5). _[Source: epics.md#Story-1.8 "When any post type is rendered or queried, Then `comments_open`/`pings_open` resolve to false"; architecture.md AD-5a guardrail "Default commenting stays disabled site-wide (feature 1.8: `comments_open` / `pings_open` → false)" (lines 404–408); project-context.md "WP comments are disabled site-wide", "❌ Use WP comments for engagement — they're disabled site-wide"]_

2. **Comment support is removed from all post types and existing open threads are closed** — Given WordPress registers `comments` (and `trackbacks`) post-type support, when `ink-core` boots, then `comments` and `trackbacks` support is removed from every registered post type (`post`, `page`, and any CPT) so the editor comment-discussion panel and the front-end comment form never render, and queries for new comments are turned away. This complements AC-1 (the filters are the query-time guard; support removal is the registration-time guard) — both are required so neither the editor UI nor a stale `comment_status=open` post can surface the free-form thread. _[Source: project-context.md "WP comments are disabled site-wide" / non-goal §13 "no WP comments"; architecture.md AD-5a "the 'no WP comments' non-goal (§13) is about the free-form UX" (lines 406–408); WordPress `remove_post_type_support`, `init` timing]_

3. **The comment admin/UI surface is hidden where appropriate** — Given the WP admin and front end, when comments are disabled, then: the admin **Comments** top-level menu is removed; the admin-bar **Comments** node is removed; the dashboard **Recent Comments / Activity** comment widget is suppressed; and front-end comment **feeds** (`/comments/feed/`) are turned off (redirected to the post or 404'd via the feed-link/template guards). The core **Comments admin list screen remains reachable as the moderation surface** for the sanctioned programmatic custom comment types per AD-5a (so moderator/`ink_reaksie` rows can still be moderated) — i.e. hide the *menu chrome and free-form entry points*, do not unregister the comments subsystem. _[Source: architecture.md AD-5a "Moderation rides the core Comments admin screen — English chrome, acceptable per the admin-language split" (lines 412–414), guardrail "Default commenting stays disabled" (lines 404–408); project-context.md admin-language split (WP-core chrome English); WordPress `admin_menu`, `wp_before_admin_bar_render`/`admin_bar_menu`, `wp_dashboard_setup`, `feed_links*`]_

4. **Lives in `ink-core` (Engagement module), wired through the 1.7 Kernel module seam — not the theme** — Given the three-layer separation and the 1.7 scaffold, when this behaviour is added, then it is implemented inside `ink-core` in the **Engagement module** (the architecture's owner of reading-engagement behaviour and of the comment substrate per AD-5a), registered via the existing `Ink\Kernel\Plugin::addModule()` seam and dispatched from the Kernel's `init` `registerModules()` call — **not** via a parallel `add_action`/`add_filter` mechanism and **not** in `ink-foundation/functions.php` or any template. The Engagement `Module::register()` (a documented no-op at 1.7) is fleshed out to register the comment-disable hooks (delegating to a dedicated `Ink\Engagement\Comments` collaborator so `Module` stays a thin bootstrap, mirroring the Kernel `Plugin`→collaborators house style). `ink-core.php` registers the Engagement module with the Kernel at boot. _[Source: architecture.md AD-1 "each module = a directory with its own bootstrap registering its own hooks; the plugin main file loads Kernel, then each module's bootstrap" (lines 141–143), AD-5a "Engagement module owns this behind its facade" (line 421), module diagram (line 134); 1-7 scaffold `Plugin::addModule()`/`registerModules()` seam, reserved `src/Engagement/Module.php`; project-context.md three-layer "No business logic in the theme"]_

5. **The sanctioned programmatic seam is preserved and documented — `wp_insert_comment` for the custom types is NOT blocked** — Given AD-5a's two sanctioned programmatic custom comment types (`ink_reaksie` for Gemeenskapsreaksies, Epic 7; `ink_moderator_terugvoer` for moderator feedback, Story 12A.5), when the disable layer is installed, then it disables only the **public/free-form** path (`comments_open`/`pings_open`, support removal, UI chrome) and **does not** interfere with programmatic `wp_insert_comment()` writes (which bypass `comments_open` by design) — so Engagement (Epic 7) and Challenges/12A.5 can still write their structured rows. A single documented extension point, the `ink_comment_open_exception` filter (default `false`), is provided: the `comments_open`/`pings_open` callbacks return `false` **unless** `apply_filters('ink_comment_open_exception', false, $post_id, $context)` returns `true`, so a future reserved ink comment-type flow can opt a specific context back in without re-enabling site-wide commenting. This story registers **no** exception (default stays `false`); 12A.5 / Epic 7 are out of scope here. The custom comment **types are not registered** here either — only the disable layer + the seam. _[Source: architecture.md AD-5a "two sanctioned programmatic custom comment types (`ink_reaksie`, `ink_moderator_terugvoer`)" (line 377), "Only programmatic `wp_insert_comment` with the custom type writes these rows" (lines 405–406), "the same sanctioned programmatic custom-type pattern … does not re-enable native WP comments" (lines 393–398); epics.md#Story-1.8 "the moderator-feedback comment type (12A.5) remains the only programmatic exception", #Story-12A.5 "via `wp_insert_comment` (not a re-enabled WP comment)"; project-context.md "Reading engagement lives in ink-core … Structured responses (Gemeenskapsreaksies)"]_

6. **`ink-core`'s own structured-response storage is NOT treated as a WP comment to disable** — Given that Gemeenskapsreaksies, line highlights+reactions, suggested reads, reading list, ratings and pinned works live in `ink-core` (custom tables and — per AD-5a — the comment substrate written programmatically), when the disable layer runs, then it must not delete, block, or disable any of those stores or their write paths. The disable layer touches **only** WP's public commenting surface (open flags, post-type support, comment UI/feeds). It adds no `$wpdb` query, no table mutation, and no removal of comment rows. _[Source: project-context.md "Reading engagement lives in ink-core, not WP comments … Structured responses, line highlights + reactions, suggested reads, reading list, ratings & reviews, pinned works"; architecture.md AD-5/AD-5a (lines 353–429) custom tables + comment substrate owned by Engagement]_

7. **Strict, prefixed, escaped, i18n, in-scope — and statically verifiable** — Given every PHP file this story creates or edits, when verified, then: each `.php` starts `<?php` then `declare(strict_types=1);` then the `Ink\…` namespace, with the `defined('ABSPATH') || exit;` guard; all classes are `Ink\`-namespaced PascalCase with camelCase methods; every hook callback and the exception filter are `ink_` snake_case where they are global/hook surface (callbacks live as camelCase methods invoked via first-class callables, the filter **tag** is `ink_comment_open_exception`); any user-facing string surfaced (none are expected — the disable layer emits no copy) would be `ink-core`-domain Afrikaans-source, escaped at output; no raw superglobals; no business logic leaks into the theme (`ink-foundation/functions.php` untouched); the module is actually wired via the Kernel seam (grep-verifiable `addModule(... Engagement\Module ...)`); the filters target `comments_open`/`pings_open` and return `false`. **`php -l` and Pest execution are deferred to Story 1.11** (no PHP binary / no harness in-repo — same precedent as Stories 1.1–1.7); structural python3 checks substitute, and the Pest unit test is authored ready-to-run in the conventional `tests/Unit/Engagement/` location. _[Source: project-context.md "declare(strict_types=1) in all ink-core PHP", "Prefix everything ink_/Ink\", escape-on-output, i18n Afrikaans-source, "No business logic in the theme", "no raw $_POST/$_GET"; architecture.md Naming/Structure patterns (lines 827–852); 1-7 deferred-`php -l`/harness-to-1.11 precedent + static-verification discipline]_

## Dev Notes

### What this story is (and is NOT)

- **IS:** a thin, hook-based **comment-disabling layer in `ink-core`'s Engagement module** that (1) forces `comments_open`/`pings_open` to `false` for every post type/query via late-priority filters, (2) removes `comments`/`trackbacks` post-type support so the editor panel and front-end form never render, (3) hides the comment **admin/UI chrome** (Comments menu, admin-bar node, dashboard widget) and front-end comment **feeds**, and (4) exposes a **single documented `ink_comment_open_exception` filter** (default `false`) as the sanctioned re-enable seam. It is wired through the **existing 1.7 Kernel module seam** (`Plugin::addModule()` → `registerModules()` on `init`) by fleshing out `Engagement\Module::register()` to register a new `Ink\Engagement\Comments` collaborator, and registering the Engagement module in `ink-core.php`. _[Source: epics.md#Story-1.8; architecture.md AD-5a guardrails (lines 403–422), AD-1 module wiring (lines 141–143); 1-7 scaffold]_
- **IS NOT:** registering the custom comment types `ink_reaksie` / `ink_moderator_terugvoer` (Engagement Epic 7 + Story 12A.5); building Gemeenskapsreaksies, line highlights+reactions, suggested reads, reading list, ratings, pinned works (Epic 7); implementing the moderator-feedback ingestion / writer display toggle (Story 12A.5); the structured-response REST write endpoint (`/ink/v1/gemeenskapsreaksie`, Epic 7); the locale/admin-language mechanism (Story 1.10); the test **harness/runner** (Story 1.11 — this story authors the test file but does not stand up Pest/CI). It also does **not** delete or block any `ink-core` engagement store, does **not** unregister the comments subsystem, and does **not** touch the theme. _[Source: epics.md Stories 1.8/1.10/1.11, Epic 7, Story 12A.5; project-context.md three-layer]_

### Why the Engagement module (placement decision)

The architecture's module diagram places **Engagement** under Content alongside Social/Sponsors/Discovery (line 134), and AD-5a states the **"Engagement module owns this behind its facade"** (line 421) for the comment substrate, explicitly binding **feature 1.8** (`comments_open`/`pings_open` → false) to the same module that will later own Gemeenskapsreaksies and the moderator-feedback type. Co-locating the disable layer with the module that owns the *sanctioned* comment writes is correct: the one module that knows the seam (which programmatic types are allowed) owns both halves — closing the public path and (later) opening the programmatic ones. The reserved `src/Engagement/Module.php` from 1.7 is the intended home; its `register()` no-op is the documented extension point this story fills. A Kernel-level placement was considered and rejected — comment policy is an Engagement-domain concern, not a cross-cutting Kernel concern (Kernel holds enums/bootstrap/schema-registry/i18n only), and putting it in Kernel would split comment ownership across two layers. _[Source: architecture.md AD-5a (lines 383–422), AD-1 diagram (line 134), Kernel contents (lines 129–130); 1-7 reserved Engagement module]_

### How the 1.7 Kernel module seam works (use it, don't reinvent)

From `src/Kernel/Plugin.php` (1.7): `boot()` is an idempotent singleton wired on `plugins_loaded`; `run()` wires `I18n::load()` on `init` and `registerModules()` on `init`. `addModule(string $id, Module $module): Plugin` stores a module keyed by id; `registerModules()` loops them and calls `$module->register()` — currently a **no-op because no module is registered**. The `Ink\Kernel\Module` interface types the contract (`register(): void`). **This story's wiring:** in `ink-core.php`, after `add_action('plugins_loaded', Kernel\Plugin::boot( ... ))`, register the Engagement module so it is in the map before `init` fires — e.g. boot the instance and `addModule('engagement', new \Ink\Engagement\Module())`, or hook the `addModule` call on `plugins_loaded` at a priority after `boot`. The cleanest match to the scaffold: call `\Ink\Kernel\Plugin::instance()->addModule('engagement', new \Ink\Engagement\Module())` inside a `plugins_loaded` callback (the same hook `boot` uses), since `instance()` boots-or-returns the singleton and `registerModules()` runs later on `init`. Confirm ordering: `addModule` must run **before** the `init` dispatch — `plugins_loaded` precedes `init`, so registering on `plugins_loaded` is safe. Do **not** add a parallel `add_filter('comments_open', …)` directly in the bootstrap. _[Source: 1-7 `src/Kernel/Plugin.php` boot/run/addModule/registerModules; architecture.md AD-1 lines 141–143]_

- The exception seam is the **only** place the close result can flip; it defaults to `false`, so behaviour is "closed everywhere" until a later story filters a specific context.
- **Admin-bar removal** can use `admin_bar_menu` at a late priority (`$wp_admin_bar->remove_node('comments')`) — pick whichever is cleanest; both are acceptable hook-based approaches.

### The sanctioned-exception seam (CRITICAL — 12A.5 + Epic 7)

AD-5a sanctions **two** programmatic custom comment types written via `wp_insert_comment`: `ink_reaksie` (Gemeenskapsreaksies, Epic 7) and `ink_moderator_terugvoer` (moderator feedback, Story 12A.5). The epics AC for 1.8 names 12A.5 as "the only programmatic exception"; the architecture additionally sanctions `ink_reaksie` under the identical pattern. **Both are preserved by the same mechanism**: WordPress's `wp_insert_comment()` does **not** check `comments_open`/`pings_open` — those guard the *public comment form / new-comment-from-front-end* path, not direct programmatic inserts. So forcing the open flags false and removing support **does not** block the programmatic writes the structured stores depend on. That is the inherent seam.

On top of that inherent behaviour, this story adds **one explicit, documented extension point** — the `ink_comment_open_exception` filter (default `false`) consulted inside `closeComments()`. A later story that genuinely needs a UI-path context re-opened for a reserved ink comment-type can filter it to `true` for that narrow context **without** re-enabling site-wide commenting. This story registers no exception. The class docblock must spell out: (a) what the seam is for, (b) that `wp_insert_comment` is already unaffected, (c) that re-enabling site-wide commenting is never the intent.
