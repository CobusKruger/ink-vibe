## Dev Notes

### What this story is (and is NOT)

- **IS:** the **Lidmaatskap renewal SECTION** — a self-contained, reusable FSE block pattern (`ink-foundation/lidmaatskap-hernu`) where a logged-in lid chooses a 1/6/12-month term to RENEW (extend access) by buying another fixed term via the 4.2 PayFast flow. Plan rows (term/price/`price_display`/sellability/`purchase_url`) come from `ink-core` (`Ink\Entitlement\RenewalPresenter`, which REUSES the 4.4 `PlanPresenter` shaping + the 4.1 registry + 4.2 `purchaseUrl()`) through a `class_exists`-guarded theme bridge. Plus an INTERIM `page-my-profiel-lidmaatskap` host template so the section is verifiable before the Epic-9 My Profiel container exists. _[epics.md#Story-4.5; FR-8; page-map.csv line 11]_
- **IS NOT:** the My Profiel page / tab framework / BuddyPress nav (**Epic 9**, Stories 9.1/9.4); the Lidmaatskap public page (**4.4** — built); the checkout / PayFast flow (**4.2** — built, just linked via `Api::purchaseUrl()`); store-UI suppression (**4.6**); Afrikaans status messages (**4.7**); lifecycle emails (**4.8**); **auto-renew / recurring / any discount (4.9–4.11)**; any new plan business rule (REUSE 4.1 registry + 4.4 PlanPresenter); any `Ink\Tiers` / Gradering coupling.

### ⭐ The decided approach — a reusable renewal SECTION pattern + a `RenewalPresenter` that REUSES the 4.4 read-model, hosted on an interim page until Epic 9.4

**The architecture question: where does the renewal tab/section live, and how is it verifiable before My Profiel exists?**

1. **WooCommerce / WooCommerce Memberships** own the sellable products + admin-editable price (AD-5). Untouched.
2. **`ink-core` (`Ink\Entitlement`)** owns the plan seam: the 4.1 registry + the 4.2 purchase hand-off + the 4.4 `PlanPresenter` row shaping, all behind the `Api` facade. This story adds ONE thin read-model, `RenewalPresenter`, that REUSES `PlanPresenter`'s shaping (delegates to it) so the renewal rows are the same closed 1/6/12-term shape with the same runtime price + `purchase_url` — **no plan/price logic is duplicated**, no discount field is added.
3. **`ink-foundation` (theme)** is presentation-only: a renewal SECTION pattern (`lidmaatskap-hernu`) that reads rows via the `class_exists`-guarded `ink_foundation_renewal_plans()` bridge (the `ink_foundation_membership_plans()` precedent) and renders them token-only + structurally locked. A guarded `ink_foundation_is_member_logged_in()` gate decides whether to show the renew section or a "Meld aan om te hernu" fallback — no entitlement logic in the theme.

**Why a section pattern, not the My Profiel page.** The My Profiel container (`page-my-profiel`, the tab framework, the `membership` tab) is **Epic 9 Story 9.4** (+ BuddyPress config 9.1), both `backlog`. Building it here would conflate epics (project rule). Delivering 4.5 as a self-contained **section pattern** means: (a) it is the exact unit the future 9.4 Lidmaatskap tab embeds (`wp:pattern` / a BuddyPress tab render of the slug); (b) it is verifiable in isolation NOW via the interim `page-my-profiel-lidmaatskap` host; (c) zero rework when 9.4 lands — 9.4 just renders the slug inside the real tab and the interim host is retired. **Dependency documented: 4.5 delivers the renewal section; Epic 9.4 owns the My Profiel container that hosts it as the Lidmaatskap tab.**

**Why no new state-changing ink-core endpoint.** "Renew" is a LINK to the WooCommerce checkout for the chosen term (`Api::purchaseUrl()` — the 4.2 seam). The WC checkout is the WC flow's own nonce surface (project-context.md says exactly this); the theme builds no card form, posts to no ink-core handler, and references no PayFast credential. So there is no ink-core nonce/cap surface to add here — the renew action is a navigation, and the purchase activation is 4.2's already-built (and nonce-bearing) WC flow.

- **REUSE, don't duplicate, the plan/price logic.** `RenewalPresenter` delegates to the 4.4 `PlanPresenter` (and thus the 4.1 registry + 4.2 `purchaseUrl()`).
