## 3. What was harder â€” challenges & lessons

- **Cross-command interaction is the real risk in a multi-command epic (R16-1, HIGH).** `PostReclassifier` (post_type `post`/`inkpols`) and `LibraryTrainingMigrator` (`/biblioteek/`Â·`/opleiding/` posts) overlapped: a flat post under a kept prefix was in *both* result sets, so whichever ran second re-recorded `SOURCE_URL_META` with the already-changed permalink and the 301 was silently lost â€” and CLI run order is unconstrained. **Lesson:** when several once-off commands mutate the same rows, make them **disjoint by construction** (PostReclassifier now skips kept-prefix posts) and make shared side-effects **write-once** (recordSourceUrl), so correctness does not depend on run order. *(New action item.)*
- **A migration must distinguish a live entity from a dead one (R16-2, HIGH).** `RedirectGenerator` built a 301 *target* from `get_permalink()` without checking post status â€” a trashed/draft post yields a `?p=`/trash URL, i.e. a bogus redirect. **Lesson:** any URL a migration emits must come from a **published** source; guard `currentPermalink` to publish-only. The runtime serve handler also gained an `is_404()` guard so a later-created page reusing an old path is never hijacked (R16-4).
- **A "strip the tags" regex is a data-mutation, so its edge cases corrupt persisted content (R16-3).** `[^\]]*` truncates a `[vc_*]` tag at a `]` inside a quoted attribute, leaving orphaned text in the post body that is then saved. **Lesson:** when a regex *rewrites stored content*, it must tolerate the quoting the source language allows â€” the quote-aware pattern handles `]` inside `"â€¦"`/`'â€¦'`. Treat content-rewriting regexes with the same rigour as SQL.
- **"Never guess" governs the default path, not authoritative input.** The Edge Hunter flagged that 16.3 imports an explicit `meester` from the CSV despite Meester being auto-promotion-manual-only. On reflection this is **correct**: the tier spreadsheet is the authoritative record of a writer's *current* grade, so an explicit value is not a guess â€” `isManualOnly` constrains only the auto-promotion engine. **Lesson:** distinguish "don't fabricate a value when input is missing/ambiguous" (the real rule) from "reject a valid authoritative value" (not the rule). Dismissed with rationale, not patched.
- **The once-off pattern has no partial-resume â€” and that's an accepted trade.** Every command `markDone()`s after its loop with no per-item progress marker, so a process killed mid-loop restarts from scratch. This matches the house pattern and is safe (every op is idempotent on full restart), but it is a known limitation worth stating rather than discovering. **Lesson:** for destructive once-off scripts, document the recovery model (full restart, not resume) explicitly.
- **phpcs `disable`/`enable` is lexical, not scoped (recurred).** A `phpcs:enable` placed inside an early-return block re-enabled the sniff before the later interpolated-SQL line, surfacing 2 errors in 16.9. **Lesson:** a `phpcs:disable` for a `$wpdb` method must span lexically from the first to the last guarded call; never place an `enable` inside a conditional branch in the middle.
