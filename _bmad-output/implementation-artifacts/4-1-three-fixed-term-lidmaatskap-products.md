# Story 4.1: Three fixed-term lidmaatskap products

Status: done

## Story

As a betaalde lid,
I want to choose a fixed-term lidmaatskap,
So that I can pay for the access term that suits me. (FR-4)

## Acceptance Criteria

1. **Three fixed-term lidmaatskap plan slots are offered — R60 / 1 maand, R300 / 6 maande, R600 / 12 maande — with the term lengths fixed at 1 / 6 / 12 months and NO auto-renew at launch.** Given the WooCommerce products / WooCommerce Memberships store (the platform plugins that OWN the sellable products), when the launch plans are enumerated through `ink-core`, then exactly three plan slots exist, each carrying a **fixed term length** drawn from the closed set {1, 6, 12} months, mapped to its WooCommerce product, with the **launch prices R60 / R300 / R600 owned by WooCommerce admin** (the configured product price), and **none of them is recurring / auto-renewing** — the registry exposes no recurring flag and the seam asserts the non-recurring launch invariant. The term lengths are the part `ink-core` holds (the AC says they "stay 1/6/12"); the prices are read from WooCommerce at runtime, never baked into PHP. _[Source: epics.md#Story-4.1 AC ("R60/1mo, R300/6mo, R600/12mo are offered (terms stay 1/6/12), no auto-renew at launch"); epics.md line 41 (FR-4 "Membership products — configurable price & term"); architecture.md AD-5 (WooCommerce/Memberships is the platform store; each entity uses "platform store where one exists"); architecture.md AD-1 (Entitlement = "WC Memberships seam"); project-context.md ("WooCommerce + WooCommerce Memberships … do not reimplement"); afrikaans-terms.md line 44 (`membership_plan` "aansluitingsopsie — Drie vaste termyne: 1 maand / 6 maande / 12 maande")]_

2. **A redakteur can change the price/term or add/retire a plan via the WooCommerce admin — there are NO hardcoded price/term VALUES baked into `ink-core` logic.** Given a redakteur editing a product in WooCommerce admin, when they change a price, change a term, or retire/add a plan, then the change takes effect WITHOUT an `ink-core` code change: the **price is resolved from the WooCommerce product at runtime** (via the WC product API behind the seam — never a PHP literal), and a plan's presence/linkage is driven by the WC product mapping (config), not by literal `R60`/`R300`/`R600` amounts hardcoded in the registry's logic. The only thing `ink-core` owns as code is the **fixed term-length value set** (the `1/6/12`-month enum the AC explicitly says stays fixed) and the **product-ID mapping seam** (admin/`.env`-resolvable), so editorial price/plan changes are a WooCommerce-admin operation. _[Source: epics.md#Story-4.1 AC ("a redakteur can change price/term or add/retire a plan via WooCommerce admin (no hardcoded values)"); FR-4 ("configurable price & term"); project-context.md ("Model fixed value sets as enums; the string/int is the persisted value; never duplicate literals" — the term set is the enum, the price is NOT a literal); architecture.md AD-4 (".env config keeps … environment flags out of code"; premium plugins assembled at build); AD-1 (Entitlement is the WC Memberships seam — the runtime lookup boundary)]_

3. **No vanity discount / savings framing appears at launch.** Given the three plan slots, when any plan label, description, or the data the registry exposes for the Lidmaatskap page (4.4) / renewal UI (4.5) renders, then there is **no "X% off", no "save R…", no "best value", no per-month-equivalent savings comparison** — the longer terms are simply longer terms at their configured price. The registry carries no discount/savings field, and no such framing is authored in any `ink-core` label. (Genuine recurring discount is post-launch — Story 4.11.) _[Source: epics.md#Story-4.1 AC ("no vanity discount/savings framing appears at launch"); epics.md#Story-4.5 AC ("no discount/savings labels at launch"); epics.md#Story-4.11 ("a genuine recurring discount … no vanity '%-off' framing"; post-launch); §14.5 amended 2026-06-20]_

## Dev Notes

### What this story is (and is NOT)

- **IS:** a config-driven, single-source **plan-registry seam** in the `Ink\Entitlement` module that DESCRIBES the three launch lidmaatskap plan slots — each a fixed `LidmaatskapTerm` (1/6/12 months) mapped to a WooCommerce product, with the **price resolved from WooCommerce at runtime** (admin-editable, never a PHP literal) — exposed through a small `Entitlement\Api` facade for the 4.3 gate / 4.4 page / 4.5 renewal to consume. It is the FIRST story to populate the reserved `Ink\Entitlement` module and wire it through the Kernel `addModule` seam. _[epics.md#Story-4.1; architecture.md AD-1/AD-4/AD-5; FR-4]_
- **IS NOT:** the PayFast purchase flow (**4.2**); the entitlement gate `can_submit()` (**4.3** — still reserved); the Lidmaatskap page / pricing-table UI (**4.4** — theme); the renewal UI (**4.5**); WooCommerce store-UI suppression (**4.6**); status messaging (**4.7**); lifecycle emails (**4.8**); **auto-renew / recurring / any discount** (**4.9–4.11**, post-launch — the registry must not model them); any `Ink\Tiers` / Gradering coupling; any reimplementation of WooCommerce or WooCommerce Memberships.

### ⭐ The decided approach — extend `Ink\Entitlement` (NOT a new Membership module)

Per the task guidance and architecture.md AD-1, the module map names the seam **"Entitlement (WC Memberships seam → can_submit())"** — membership plans/products are squarely in the membership/entitlement domain, and the reserved `Ink\Entitlement\Module` stub already declares it "will own the WooCommerce Memberships seam." A separate `Ink\Membership` module is NOT named in AD-1 and would fragment the domain (the 4.3 `can_submit()` gate must read the same membership concept). **Decision: extend `Ink\Entitlement`** — add the plan registry + term enum + WC price resolution + facade there, populating the stub. The 4.3 `can_submit()` gate stays reserved in the same module for its own story.

The shape:
- **WooCommerce / WooCommerce Memberships** own the sellable products + the price (admin-editable) — the platform store (AD-5). `ink-core` does NOT reimplement them and holds NO price literal.
- **`ink-core` (`Ink\Entitlement`)** owns the **business seam**: the fixed term-length value set (`LidmaatskapTerm` enum — the part the AC says "stays 1/6/12"), the product-ID mapping (config/`.env`-resolvable), the runtime price resolution behind a `function_exists( 'wc_get_product' )` guard, and the `Api` facade.

### ⚠️ Guardrails (prevent disasters)

- **No hardcoded price/term VALUES (FR-4 "configurable").** The enum holds the fixed term LENGTHS (1/6/12 — explicitly fixed by the AC); the PRICE is resolved from the WooCommerce product at runtime. No `60`/`300`/`600` literal in `ink-core` logic.
- **No auto-renew, no recurring, no discount/savings framing at launch** (4.9–4.11 are post-launch). The registry exposes no recurring flag and no discount field; no vanity "%-off"/"save R…"/"best value" copy.
- **THE conflation rule (AD-1).** Lidmaatskap entitlement ⟂ writer Gradering. **Zero `use Ink\Tiers`** in `src/Entitlement/`.
- **Do not reimplement WooCommerce / WC Memberships.** Read product price via the WC product API behind a `function_exists()` guard; degrade gracefully when WC is absent (unit suite has no WC).
- **Afrikaans-only labels via the registry**, sentence case, no English `.mo`, no AI-invented prose (`[NEEDS HUMAN AFRIKAANS]` for plan descriptions / CTA copy).
- **strict types + `ABSPATH` guard + `Ink\Entitlement` namespace + enum for the fixed set**; tests run and pass before done.
- **Wire through the Kernel seam** — `addModule( 'entitlement', … )` in `ink-core.php` + `Module::register()`; no logic in the bootstrap, no theme/`functions.php` logic.
