## Story

## Acceptance Criteria

1. **A Lidmaatskap page renders as an assembly-only, pricing-table FSE pattern + page template that shows plans, benefits, FAQ, and a purchase CTA.** Given the Lidmaatskap page (`page-lidmaatskap` template assembling the `ink-foundation/lidmaatskap` pattern), when it renders, then it presents — in order — a page intro, a **pricing table** comparing the three fixed-term aansluitingsopsies (1 maand / 6 maande / 12 maande), a **benefits** section, an **FAQ** section, and a **purchase CTA** per plan that starts the WooCommerce/PayFast purchase (4.2). It is composed entirely from existing core/registered blocks (`core/group`, `core/columns`, `core/heading`, `core/paragraph`, `core/buttons`/`core/button`, `core/separator`, `core/details` for the FAQ) plus the established INK block styles (`is-style-card` / `is-style-emphasis` / `is-style-pill`) — no heavy new component is invented. _[Source: epics.md#Story-4.4 AC ("the Lidmaatskap page (assembly-only, pricing-table pattern) … shows plans, benefits, FAQ, and CTA"); page-map.csv line 14 (`lidmaatskap` → `page-lidmaatskap`, `template+patterns`, `pricing-table;buttons;faq`, `data_source: woocommerce memberships`, `status: assembly-only`); patterns/README.md "Authoring rules" + "Compose, don't re-pattern"; FR-7 line 44/167]_

2. **Quality Gate A — every colour, spacing, and type value maps to a `theme.json` token; no hardcoded hex/px/raw font size appears in the pattern markup, the template, or any added style.** Given the pattern + template + any block-style CSS this story touches, when they are authored, then every value resolves to a `var:preset|*` / `var:custom|*` token or an existing block-style class — there is **zero** literal `#hex`, `px`, or unnamed font size. (If a pricing-table-specific block style is added, its CSS uses only `--wp--preset--*` / `--wp--custom--*` custom properties, mirroring `functions.php`'s existing `is-style-card`/`is-style-emphasis` registrations.) _[Source: project-context.md Gate A ("No hardcoded colours, spacing, or unnamed type sizes … everything maps to theme.json tokens"); patterns/README.md "Gate A — tokens only"; functions.php `ink_foundation_register_block_styles()` (token-only inline_style precedent); theme.json color/spacing/fontSize/shadow/custom presets]_

3. **No business logic in the theme: plan data (term label, price, sellability, purchase URL) is sourced from `ink-core` via a presentation seam — the theme never hardcodes prices, never iterates plan business rules, and never re-queries WooCommerce directly.** Given the three-layer rule, when the pattern needs plan data, then it reads it through a thin, `class_exists`-guarded theme bridge (`ink_foundation_membership_plans()`) that delegates to the `Ink\Entitlement\Api` facade (4.1 registry + 4.2 `purchaseUrl()`), which resolves the price from the WooCommerce product at runtime and the checkout URL from the WC PayFast hand-off. The price (R60/R300/R600 at launch) is **never** a literal in the theme; the term labels come from the `ink-core` terminology registry; the per-plan purchase CTA links via `Api::purchaseUrl()`. When `ink-core` is inactive or a plan is not sellable, the seam degrades gracefully (the slot renders its static label/benefit copy without a live price/CTA — never a fatal, never an invented endpoint). The plan-row SHAPING (term + price + purchase URL + availability per slot) lives in a new `ink-core` read-model (`Ink\Entitlement\PlanPresenter`), so the theme holds no plan-iteration logic at all. _[Source: project-context.md three-layer ("No business logic in the theme … no … logic in templates or functions.php"); "Hook, don't edit"; architecture.md AD-1 (Api facade = only cross-module surface) / AD-10 (terminology + binding seams bridge ink-core data into static presentation); 4-1 story (price resolved from WooCommerce at runtime, no literal; `Api::plans()`/`priceFor()`/`isAvailable()`); 4-2 story (`Api::purchaseUrl()` hands off to WC checkout / PayFast); functions.php `ink_foundation_term()` / `ink_foundation_social_login_*` (the `class_exists`-guarded presentation-bridge precedent)]_

4. **Quality Gate D — all copy is curated Afrikaans (sentence case), from `docs/ui-copy-translations.md` and/or the terminology registry; no English leakage, no AI-invented Afrikaans, no lifted Lovable placeholder text. Un-curated member-facing prose is a clearly-marked `[NEEDS HUMAN AFRIKAANS]` placeholder, flagged — never invented.** Given every string the page surfaces, when it renders, then: the plan term labels come from the registry (`term_1_month` / `term_6_months` / `term_12_months`), the aansluitingsopsie noun from `membership_plan`; the per-plan member-facing descriptions and the page CTA copy are the `[NEEDS HUMAN AFRIKAANS]` rows already reserved in `ui-copy-translations.md` (the 4.1 "Aansluitingsopsies" subsection, lines ~550–555) and stay clearly-marked placeholders until human copy lands; the benefits + FAQ copy is sentence-case Afrikaans drawn from approved glossary terms and the Lidmaatskap-blad section, with any un-curated string flagged `[NEEDS HUMAN AFRIKAANS]`. No English placeholder is lifted from the Lovable mock. _[Source: project-context.md Gate D ("front end entirely Afrikaans … sentence case … No AI-generated Afrikaans … Controlled-vocabulary UI labels come from the ink-core terminology registry … never inline a glossary label"; "Never lift copy from the Lovable mockup"); ui-copy-translations.md Lidmaatskap-blad + "Aansluitingsopsies — drie vaste termyne" ([NEEDS HUMAN AFRIKAANS] rows, lines ~544–555); afrikaans-terms.md (lidmaatskap / aansluitingsopsie / term-length labels); 4-1 story AC-4 (plan PROSE stays `[NEEDS HUMAN AFRIKAANS]`)]_

## Dev Notes

### What this story is (and is NOT)

- **IS:** the **Lidmaatskap PAGE** — an assembly-only, pricing-table FSE block pattern (`ink-foundation/lidmaatskap`) + a page template (`page-lidmaatskap.html`) in `ink-foundation`, composing existing core blocks + INK block styles to show **plans, benefits, FAQ, and a purchase CTA**. Plan data (term/price/sellability/purchase-URL) is sourced from `ink-core` (`Ink\Entitlement\Api`, the 4.1 registry + 4.2 purchase hand-off) through a thin, `class_exists`-guarded theme bridge fed by a minimal `PlanPresenter` read-model — **no business logic, no price literal, no WooCommerce re-query in the theme.** Also closes the 4.1-deferred vanity-savings-copy cleanup. _[epics.md#Story-4.4; page-map.csv line 14; FR-7]_
- **IS NOT:** the renewal UI / My Profiel Lidmaatskap tab (**4.5**); the actual checkout / PayFast flow (**4.2** — built, just linked via `Api::purchaseUrl()`); WooCommerce store-UI suppression (**4.6**); Afrikaans status messages (**4.7**); lifecycle emails (**4.8**); auto-renew / recurring / any discount (**4.9–4.11**); any new plan business rule (consume the 4.1 registry, never extend it); any `Ink\Tiers` / Gradering coupling.

### ⭐ The decided approach — PHP pattern + a `class_exists`-guarded theme bridge over the existing `Ink\Entitlement\Api`

The architecture question is: **how does a static block pattern surface dynamic plan data without putting business logic in the theme?** Three layers:

1. **WooCommerce / WooCommerce Memberships** own the sellable products + the admin-editable price (the platform store, AD-5). Untouched.
2. **`ink-core` (`Ink\Entitlement`)** owns the plan business seam: the 4.1 registry (`MembershipPlans` / `LidmaatskapTerm`, runtime price from WooCommerce, no literal) + the 4.2 purchase hand-off (`PurchaseActivation::purchaseUrl()`), all behind the `Api` facade. This story adds ONE thin read-model, `PlanPresenter`, that shapes the per-slot presentation row (term label + price + sellability + purchase URL) — delegating wholly to `Api`, adding no rule.
3. **`ink-foundation` (theme)** is presentation-only: a PHP pattern that reads the rows via a `class_exists`-guarded bridge `ink_foundation_membership_plans()` (mirroring the established `ink_foundation_term()` / `ink_foundation_social_login_*()` bridges) and renders them with token-only, structurally-locked core blocks. The theme iterates rows it is HANDED; it computes nothing.

**Why the PHP-bridge seam, not the Block Bindings source:** the existing AD-10 binding source (`Ink\I18n\Bindings`'s `ink/term`) exists specifically because **static `templates/*.html` cannot call PHP**. The Lidmaatskap page is a **PHP pattern** (like `auth-login.php`), so it CAN call the `function_exists`-guarded bridge directly — the same precedent the auth patterns use for `ink_foundation_social_login_buttons()`. That keeps ink-core additions minimal (one read-model + tests) and avoids registering a binding source nothing consumes. An `ink/membership-plan` Block Bindings source is the documented seam a FUTURE static-HTML consumer would use; it is intentionally deferred (Task 6 decision point) to honour "keep ink-core additions minimal."

This satisfies the three-layer rule: prices come from WooCommerce via the 4.1 seam (never the theme), plan shaping lives in ink-core (`PlanPresenter`), and the theme holds only a guarded read-through + token-only markup.

### ⚠️ Guardrails (prevent disasters)

- **No price literal in the theme.** The price is echoed from the ink-core row (`Api::priceFor()` → WooCommerce runtime) — NO `R60`/`R300`/`R600`/`60`/`300`/`600` anywhere in the pattern or template. When the price is null (WC absent / unmapped / retired), the slot renders gracefully without a live price — never a fallback literal.
- **No vanity savings framing (AC-3 / AC-5).** No "X% off", "save R…", "best value", or per-month-equivalent comparison in the pattern. Remove the deferred "Save 12%"/"Spaar 25%" rows from `ui-copy-translations.md`.
- **Gate A — tokens only.** Every value a `var:preset|*` / `var:custom|*` token or a block-style class; zero hardcoded hex/px/raw font size. Mirror `cta-band.php` / `functions.php` block-style CSS.
- **Gate D — Afrikaans, sentence case.** Labels via the registry bridge; un-curated member-facing prose (plan descriptions, FAQ answers, CTA copy) stays clearly-marked `[NEEDS HUMAN AFRIKAANS]` — do NOT invent or AI-translate; do NOT lift the Lovable English mock.
- **Block locking (NFR-6 / Story 1.6).** Structural wrappers `lock:{move,remove}`; content editable; never `templateLock:"all"` / `lock:{edit:true}`.
- **THE conflation rule (AD-1).** Lidmaatskap ⟂ writer Gradering. Zero `use Ink\Tiers` in any added ink-core file.
- **Graceful degrade.** Every theme bridge is `class_exists`/`function_exists`-guarded so the theme never fatals when ink-core is inactive; the purchase CTA degrades (no invented endpoint) when `purchaseUrl()` is null.
- **No hollow unit tests on the pattern.** Block-theme presentation is E2E/visual (Story 18.8). Author Pest only for the new `PlanPresenter` ink-core logic. Suite passes before done.
