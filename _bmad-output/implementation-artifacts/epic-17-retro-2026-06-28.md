## 2. What went well

- **Audit-first prevented harmful blanket edits.** Every story opened with a classified subagent audit (Class A = must-fix UI/live · B = preserve code IDs · C = glossary/reconcile docs that document old→new · D = dated historical snapshots) *before* any edit. This is what kept 17.3 from becoming a destructive find/replace: `ink_writer_tier`, brons/silwer/goud enums, the `FollowGraphMigration` "Vriendskappe" migration-source term, the glossary, and the consolidated-spec spelling-correction *notes* were all correctly left intact. The actual Class-A residue turned out to be three doc lines.
- **Content/staging-gated work was tracked, never faked.** No AI Afrikaans was generated (the standing rule held under pressure); no org legal status was invented (the `[regstatus]` clause was *dropped* for the confirmed generic framing, not filled with a guess); the surviving-plugin `.po/.mo/.json` and the live page-crawl were recorded as explicit pre-launch staging+human gates with runbooks, not silently dropped. The "infra + runbook + tracked gate" shape (proven in 17.2/17.4) is the honest way to close an execution story that depends on a running site + a human translator.
- **The deferred Epic-2 review debt closed cleanly (17.4).** All three flagged gaps fixed: `Terms::label()` before-`init` guard, `Terms::has()` assertions in both registrars (new `assertTermKey()`), and `Bindings::resolve()` now renders **nothing** (never the raw machine key) for a misconfigured binding. Every guard surfaces in dev/CI via `_doing_it_wrong` and never fatals in production, preserving the low-level `Terms::label()` fail-safe. Each is covered by a non-vacuous paired test (fires on a bad path, silent on the good path).
- **The leak gate became real where it can be, documented where it can't.** The static `copy:scan` subset already runs in CI; the registry guards are its new source-level catch for raw-key/blank leaks; and the live crawl + `wp i18n` layer is a documented Layer-2 runbook owned by the 18.8 CI buildout. Clear separation of what's enforced today vs. what needs a running site.
- **Verification over assumption.** The biggest risk in an "execution" epic is re-doing work that's already done or, worse, "fixing" something correct. Reading the actual code/doc state first (e.g. confirming sheet sections 1–7 were already wired, that the glossary was already complete, that live code already used the follow model) meant the commits were small, surgical, and low-risk.

## 3. What was harder — challenges & lessons

- **Brain Monkey function-stubs persist process-wide — `function_exists()` guards leak across tests (NEW codified lesson).** 17.4's first guard design used `if ( function_exists('did_action') && ! did_action('init') && function_exists('_doing_it_wrong') )`. Once *any* test defined those functions via Brain Monkey, `function_exists()` stayed true for the rest of the process, so later unmocked calls threw — 21 cross-contaminated failures. **Lesson:** for a guard that calls a WP function the mocked suite doesn't load, do NOT gate on `function_exists()`; call the function directly (as production does) and provide an **inspectable test double in `tests/bootstrap.php`** (the project's "WP doubles live in bootstrap" convention) — `did_action` defaulting to "init fired" so the 1000+ existing tests stay silent, `_doing_it_wrong` recording calls for assertion. *(Promote to the testing rules — see action items.)*
- **A test double's default value is a silent assumption (R17-2, MEDIUM).** The `did_action` double's `?? 1` fallback reported *every* unseeded hook as fired, so a future guard checking a different hook would pass spuriously. **Lesson:** a test double should default to the *conservative* state (`?? 0` = not fired) and force tests to declare what's true; only the explicitly-seeded `init => 1` reports fired.
- **Global spy state needs guaranteed reset, not trailing-line discipline (R17-1, MEDIUM).** The before-`init` test set `init=0` and relied on a *trailing* `ink_reset_guard_spies()`; if an assertion above it failed, `init=0` would leak into the shared process. **Lesson:** reset shared test state in `beforeEach`, never via a manual call at the end of a test body. (Respected the per-file Brain-Monkey lifecycle convention — reset added to each guard-test file's `beforeEach`, not a global Pest hook.)
- **"Execution/QA" epic numbers hide that the work was spread across the whole project.** Epic 17 sits at number 17, but its substance was created incrementally in Epics 1–16 (every story that needed Afrikaans left a placeholder + sheet row; the glossary was updated first each time; the loaders/registry/admin-split shipped in foundation epics). **Lesson:** a late "execution + QA" epic is mostly *verification* — budget it as an audit, and treat finding "already done, correct" as a successful outcome, not a missing deliverable. The danger is inventing work to justify the epic.
- **Carry-forward typos must actually be closed, or they recur.** The Epic-16 retro flagged `niewinsgericht`→niewinsgerig and `trep`→trap (in `ui-copy-translations.md` + `gemeenskap.php`) for 17.1. The initial 17.1 pass fixed the *translation-sheet* typos but missed these two named front-end ones; the retro's follow-through check caught it and closed it (4 occurrences fixed). **Lesson:** when a retro names *specific* source-doc fixes for the next epic, the create-story step should enumerate them as explicit ACs/tasks, not fold them into a general "fix typos" line.
- **`_doing_it_wrong` message interpolation is "output" to WPCS.** The `$key` in the guard messages tripped `WordPress.Security.EscapeOutput`; wrapping in `esc_html()` is the WP-correct fix (and required stubbing `esc_html` in the four guard-test `beforeEach`). **Lesson:** even dev-only diagnostic strings that interpolate a variable need escaping under WPCS.
