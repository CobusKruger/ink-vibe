# Story 4.2: Front-end PayFast purchase flow

Status: review

## Story

As a lid,
I want to buy and self-activate a lidmaatskap via PayFast in ZAR,
So that I become a betaalde lid with no staff action. (FR-5, UJ-2)

## Acceptance Criteria

1. **A successful PayFast payment self-activates the lidmaatskap ‚Äî automatically, with NO manual EFT / admin step (this replaces the UJ-2 manual activation).** Given a lid who completes an off-site PayFast payment in ZAR for a Story-4.1 plan and is returned to the site, when the WooCommerce / WooCommerce Memberships platform plugins grant / activate the membership (their job, not `ink-core`'s), then `ink-core` REACTS to the membership-activated event and confirms the lid is now a betaalde lid ‚Äî no redakteur has to flip a status, send an EFT confirmation, or run any admin step. `ink-core` does NOT flip the membership status itself (WC Memberships owns the status); it hooks the documented WC Memberships activation transition and performs its own side effects. The activation handler fires for a genuine grant‚Üíactive transition and is inert otherwise (no double-fire on a no-op status write, no fire on a transition that is not *into* `active`). _[Source: epics.md#Story-4.2 AC ("Given a successful PayFast return, when payment completes, then the lidmaatskap activates automatically (no manual EFT/admin step) and a thank-you/activation email is sent (4.8)"); epics.md line 42 (FR-5 "Self-service PayFast purchase", P0); epics.md#Epic-4 overview ("replacing manual EFT activation (UJ-2)"); architecture.md AD-3/AD-9 addendum ("WC Memberships lifecycle ‚Üí Action Scheduler ‚Ä¶ the Entitlement module gains a notify responsibility (it emits the lifecycle events Notifications consumes)"); project-context.md ("WooCommerce + WooCommerce Memberships ‚Ä¶ do not reimplement"; "Hook, don't edit"); architecture.md AD-1 ("Entitlement (WC Memberships seam ‚Üí can_submit())")]_

2. **The activation event fires the thank-you / activation email trigger exactly once, via the Notifications API (the 4.8 email copy is forward-referenced ‚Äî 4.2 wires the trigger, not the prose).** Given the membership-activated event, when the activation handler runs, then it calls `Notifications\Api::send()` ONCE for the activation/thank-you template (keyed `ink_membership_activated_email`), resolving the recipient from the activated membership's owner and passing the `{skrywer}` greeting context. The template is REGISTERED here as an Afrikaans-source `Notifications\Template` with its send toggle **OFF by default** and a clearly-marked `[WAG OP MENSLIKE KOPIE]` placeholder body ‚Äî so NO `wp_mail` actually fires until Story 4.8 lands the curated human copy and turns the toggle on. ### What this story is (and is NOT)

- **IS:** the `ink-core` business SEAM around the off-site PayFast flow, in the `Ink\Entitlement` module ‚Äî a focused `PurchaseActivation` collaborator that (a) initiates a purchase of a 4.1 plan by handing off to the WC checkout / WC PayFast gateway, and (b) REACTS to the WC Memberships activation event to confirm the betaalde lid and fire the activation/thank-you email trigger via the Notifications API. It REUSES the 4.1 plan registry. _[epics.md#Story-4.2; architecture.md AD-1/AD-3/AD-9 addendum; FR-5/UJ-2]_
- **IS NOT:** the PayFast gateway / WC checkout / ITN callback / membership-status flip (platform-owned ‚Äî `ink-core` hooks, never reimplements); the entitlement gate `can_submit()` (**4.3** ‚Äî still reserved); the Lidmaatskap page / pricing-table UI (**4.4** ‚Äî theme); the renewal UI (**4.5**); store-UI suppression (**4.6**); the Afrikaans status-messaging surface (**4.7**); the REAL activation/lifecycle email copy + expiry-warning emails + per-term toggles (**4.8** ‚Äî 4.2 wires only the activation trigger with a placeholder, toggle OFF); **auto-renew / recurring / any discount** (**4.9‚Äì4.11**, post-launch); any `Ink\Tiers` / Gradering coupling.

### ‚≠ê The decided approach ‚Äî extend `Ink\Entitlement` with a `PurchaseActivation` collaborator (NOT a new module)

AD-1 names the seam **"Entitlement (WC Memberships seam ‚Üí can_submit())"**, and the AD-3/AD-9 addendum is explicit: *"the Entitlement module gains a notify responsibility (it emits the lifecycle events Notifications consumes)."* The PayFast purchase + the membership-activation reaction are squarely Entitlement-domain. 4.1 already wired the `entitlement` module through the Kernel `addModule` seam, so 4.2 needs no `ink-core.php` change. **Decision: add a focused `PurchaseActivation` collaborator** (mirroring the `Accounts\Module ‚Üí {Registration, Onboarding, Approval}` one-collaborator-per-concern house style), wired via the existing `Entitlement\Module::register()`. A separate `Ink\Payment` / `Ink\Membership` module would fragment the domain and add a module against the ~8‚Äì12 cap (AD-1) for no benefit.

### ‚≠ê The activation HOOK decision (justified)

`ink-core` must react to "the lidmaatskap became active", not "an order was paid". The chosen hook is **`wc_memberships_user_membership_status_changed`** ‚Äî the documented WC Memberships transition action, fired as `( WC_Memberships_User_Membership $user_membership, string $old_status, string $new_status )` on every status transition, including into `active`. Why this one:

- It is **membership-specific** ‚Äî it fires precisely when the membership is granted/activated, for BOTH the PayFast-return grant and any admin/manual grant. The handler needs no order‚Üímembership lookup.
- `woocommerce_payment_complete` / `woocommerce_order_status_completed` fire for **any** order (including non-membership purchases) and would force `ink-core` to inspect the order's line items to discover whether a membership was even involved ‚Äî more coupling, more surface, and it would miss admin-granted memberships.
- The bare `wc_memberships_user_membership_saved` fires on **every** save (profile edits, note additions), not just an activation ‚Äî it would over-fire.

The handler **gates on `new_status === 'active'`** so it is idempotent: it fires only on a transition INTO active; a no-op save or a transition to any other status does nothing (no double-send). The status string `active` is WC Memberships' own value, held as a single-source `const STATUS_ACTIVE` (not a scattered literal). The hook name is `const HOOK_STATUS_CHANGED`.

This is the AD-2-clean direction too: the **publish gate** (4.3) still evaluates the end-date in SAST and is NOT changed by this notify path (AD-3/AD-9 addendum ‚Äî "the notify path does not change `can_submit()` semantics"). 4.2 only reacts to activation for the email + betaalde-lid confirmation; it does not become the entitlement authority.

### ‚öÝÔ∏è Guardrails (prevent disasters)

- **PCI / no card data (project-context.md Security).** PayFast is off-site; the WC PayFast gateway captures the card and handles the ITN callback. `ink-core` stores NO card / PAN / CVV / cardholder data (no option, no meta, no log) and captures no card fields. No PayFast merchant id / key / passphrase as a code literal ‚Äî config comes from `.env` / option (AD-4), never logged.
- **Sandbox only in tests (NFR-9).** Every WC / WC PayFast / WC Memberships function and the activation hook are mocked; the config seam resolves to sandbox; no network, no real `wp_mail`, no PayFast endpoint.
- **Self-activation is WC Memberships' job; `ink-core` REACTS.** Hook `wc_memberships_user_membership_status_changed` gated on `active`; do not flip the membership status manually.
- **One activation = one email send attempt.** The `new_status === active` gate makes the handler idempotent; the email toggle is OFF until 4.8.
- **THE conflation rule (AD-1).** Lidmaatskap entitlement ‚üÇ writer Gradering. **Zero `use Ink\Tiers`** in `src/Entitlement/`. Buying/activating a lidmaatskap never writes `ink_writer_tier`.
- **No recurring / auto-renew** (one-off fixed-term only; 4.9‚Äì4.11 post-launch). Model no recurring concept.
- **Reuse the 4.1 plan registry** ‚Äî never re-define a plan or a price; resolve the WC product via `MembershipPlans` / `Api`.
- **Afrikaans-only template via literal `__()`, toggle OFF, `[WAG OP MENSLIKE KOPIE]` body**, no English `.mo`, no AI-invented prose (`[NEEDS HUMAN AFRIKAANS]` in `ui-copy-translations.md`; Story 4.8 owns the copy).
- **State-changing front-end paths**: nonce + capability, no raw superglobal, no raw SQL.
- **strict types + `ABSPATH` guard + `Ink\Entitlement` namespace**; tests run and pass before done.
