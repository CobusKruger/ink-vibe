# Story 1.7: ink-core plugin scaffold

Status: in-progress

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an ink-core developer,
I want a bootstrapped `ink-core` plugin structure — bootstrap file, PSR-4 autoloader, the AD-1 modular-monolith source tree, a Kernel that wires hooks, activation/deactivation hooks, and text-domain loading,
so that all INK business logic has a correct, test-ready home from which Stories 1.8, 1.10, 1.11, 1.12 and every later epic build, rather than retrofitting structure after 15 epics ship.

## Acceptance Criteria

1. **The plugin activates with a well-formed bootstrap** — Given the `ink-core` plugin, when WordPress loads `wp-content/plugins/ink-core/ink-core.php`, then the file is a valid WordPress plugin: a complete plugin header (`Plugin Name`, `Description`, `Version`, `Requires at least: 7.0`, `Requires PHP: 8.3`, `Text Domain: ink-core`, `Domain Path: /languages`, `License`), `declare(strict_types=1);` on the first line after `<?php`, an `ABSPATH` guard (`defined('ABSPATH') || exit;`), and **no procedural side effects beyond registering the autoloader, the activation/deactivation hooks, and a single `plugins_loaded` bootstrap callback**. The file defines `ink_` prefixed version/path constants (`INK_CORE_VERSION`, `INK_CORE_FILE`, `INK_CORE_PATH`, `INK_CORE_URL`) — no hardcoded URLs; paths via `plugin_dir_path()`/`plugin_dir_url()`. _[Source: epics.md#Story-1.7 "the bootstrap … in place (`declare(strict_types=1)`, `Ink\` namespace, `ink_` prefixing)"; architecture.md "Hand-rolled Composer/PSR-4 scaffold … `declare(strict_types=1)`" (lines 682, 691–693); project-context.md "declare(strict_types=1) in all ink-core PHP", "no hardcoded asset URLs", "Requires PHP: 8.3"]_

2. **A PSR-4 autoloader resolves `Ink\` → `src/`** — Given the `Ink\` namespace, when an `Ink\…` class is first referenced, then it is loaded from `src/` by PSR-4 mapping (`Ink\Kernel\Plugin` → `src/Kernel/Plugin.php`, `Ink\Kernel\Tier` → `src/Kernel/Tier.php`). The scaffold ships **both**: a `composer.json` declaring `"autoload": { "psr-4": { "Ink\\": "src/" } }` (the architecture-sanctioned mechanism; CI runs `composer install` in Story 1.11) **and** a small hand-rolled `spl_autoload_register` PSR-4 loader in the bootstrap that activates **only when** the Composer `vendor/autoload.php` is absent — so the plugin is loadable in the repo today (no committed `vendor/`, no network install) and seamlessly uses Composer's optimized autoloader once CI builds it. No `require` of individual class files in the bootstrap. _[Source: architecture.md "Composer autoload `Ink\` → `src/`" (line 692), "hand-rolled Composer/PSR-4 plugin scaffold" (line 682, 691); AD-4 "core and third-party plugins are fetched at build, not vendored into VCS … Only `ink-core` + `ink-foundation` are committed" (lines 339–342); project-context.md testing "harness arrives Story 1.11"; epics.md#Story-1.11 "Pest/PHPUnit … wired in"]_

3. **The AD-1 modular-monolith source tree exists with the canonical module skeleton** — Given AD-1 (modular monolith; reject flat plugin/plugin fleet), when the scaffold is laid out, then `src/Kernel/` holds the cross-cutting kernel (bootstrap, autoload, enums, capabilities, schema registry, i18n loader) and the runtime module directories exist as **empty-but-reserved** extension points, each following the canonical skeleton `src/{Module}/Module.php` (bootstrap + hook registration) `+ Api.php` (the sole public facade) with `Rest/`, `Repository/`, `Entity/` reserved where a module will need them — created via `.gitkeep` / interface stubs **without implementing any feature**. The reserved module set mirrors the dependency-ordered epics: `Content`, `Entitlement`, `Tiers`, `Engagement`, `Social`, `Sponsors`, `Discovery`, `Submission`, `Challenges`, `Notifications`, `Forms`. _[Source: architecture.md AD-1 module structure (lines 126–143), "Canonical module skeleton — every module is identical: `src/{Module}/Module.php` … `Api.php` … `Rest/` … `Repository/` … `Entity/`" (lines 847–850); "Keep module count bounded (~8–12)" (line 160)]_

4. **The Kernel bootstrap class wires hooks (no business logic)** — Given the `Ink\Kernel\Plugin` class, when `plugins_loaded` fires, then a single `Ink\Kernel\Plugin::boot()` (idempotent singleton) runs: it loads the text domain on the correct hook and is the **one place** later stories register their module bootstraps (`Content`, `Entitlement`, …). The class is `final`, `declare(strict_types=1)`, uses PHP 8.3 idioms (constructor promotion / `readonly` where apt, first-class callables for hook callbacks), exposes a documented `add_module()` / module-registration seam, and contains **zero feature logic** — no `register_post_type`, no taxonomy, no `ink_writer_tier`, no Membership/entitlement, no comment filters, no REST routes. Methods are camelCase; the procedural surface (`ink_core()` accessor / template-tag-style helpers) is `ink_` snake_case. _[Source: architecture.md AD-1 "the plugin main file loads Kernel, then each module's bootstrap" (line 142), "Kernel (enums … bootstrap … i18n loader)" (line 129); Naming "classes PascalCase; methods camelCase; `ink_` snake_case … global/procedural WP surface only" (lines 834–838); project-context.md "No business logic in the theme" + three-layer; epics.md#Story-1.8/1.10/1.11/1.12 (out of scope)]_

5. **Activation / deactivation hooks register with safe, versioned stubs** — Given `register_activation_hook` / `register_deactivation_hook` on the main file, when the plugin is activated, then an activation handler (a) records a **DB/schema version option** `ink_core_db_version` (so later modules' `dbDelta()` migrations can compare and upgrade — the activation handler invokes the Kernel schema registry, which is **empty** at 1.7), and (b) **flushes rewrite rules** (`flush_rewrite_rules()`) as a stub for the CPTs/taxonomies that Epic 2 will register; the deactivation handler flushes rewrite rules. Activation also guards the **PHP 8.3 / WP 7.0** minimum (deactivates + `wp_die()` with an i18n Afrikaans-source message if unmet). The handlers are static, strict-typed, `Ink\` namespaced, and create **no** tables/CPTs/terms themselves (those are each module's job in later epics). _[Source: epics.md#Story-1.7 "activation hooks … in place"; architecture.md AD-1 "each custom table is created/owned/migrated by its module" (lines 425–426, 131), "dbDelta schema registry" in Kernel (line 130); AD-5 "each store owned and migrated (dbDelta) by its module"; project-context.md "Custom tables … declared via dbDelta() with the $wpdb->prefix", "Requires PHP 8.3"]_

6. **Text-domain loading is correct and Afrikaans-source-ready** — Given i18n, when the Kernel boots, then `load_plugin_textdomain('ink-core', false, dirname(plugin_basename(INK_CORE_FILE)).'/languages')` is called on the appropriate hook (`init`, per current WP guidance, invoked from the Kernel boot wired at `plugins_loaded`), the text domain string is **exactly `ink-core`**, a `languages/` directory exists (reserved; **no English `.mo` is shipped** — gettext returns the Afrikaans source under an English admin locale per §14.15), and any string emitted by the scaffold itself (the activation min-version `wp_die` message, any admin notice) is wrapped in `__()/esc_html__()` with the `ink-core` domain and authored as **Afrikaans source text** (sentence case). _[Source: epics.md#Story-1.7 "text-domain loading … in place"; project-context.md "i18n … correct text domain (ink-core/ink-foundation); Afrikaans is the source text", "ink-core ships no English .mo", admin-language split §14.15; architecture.md "Kernel (… i18n loader)" (line 129), "ink-core ships no English .mo" (lines 820–821); memory: Afrikaans is source of truth — no AI retranslation]_

7. **The conflation-rule separation is structurally reserved, not implemented** — Given THE conflation rule (`Ink\Entitlement` ⟂ `Ink\Tiers` — subscription/lidmaatskap entitlement and writer Gradering are separate, never gated on each other), when the scaffold reserves the module directories, then `Entitlement` and `Tiers` are **two distinct sibling module directories** with **no cross-reference between them** (no `use Ink\Tiers\…` in `Entitlement`, no `use Ink\Entitlement\…` in `Tiers`), each with its own `Module.php`/`Api.php` placeholder, so the Deptrac/PHPArkitect dependency-rule check (wired in CI, Story 1.11/AD-8) can later assert the absent edge on real code. The `Tier` enum (`Brons|Silwer|Goud|Meester`, backed by lowercase Afrikaans values) lives in **`Kernel`**, not in `Tiers` — so both modules and Challenges read it from the shared Kernel without creating an inter-module edge. **No entitlement, tier, promotion, or pool logic is written.** _[Source: epics.md line 24 "`Ink\Tiers` ⟂ `Ink\Entitlement`"; architecture.md AD-1 "the absence of the Entitlement⟷Tiers edge is the conflation rule … enforce in CI with a dependency-rule checker" (lines 146–151), AD-1 module diagram "Entitlement ⟂ Tiers" (lines 132–133), Kernel holds "enums: Tier {Brons|Silwer|Goud|Meester}" (line 129); AD-9 addendum "Ink\Tiers must not reference Ink\Entitlement … and vice-versa" (lines 654–656); project-context.md THE conflation rule]_

8. **Strict, prefixed, escaped, in-scope — and statically verifiable** — Given every PHP file the story creates, when verified, then: **every** `.php` file (bootstrap + every `src/**/*.php` stub) starts `<?php` then `declare(strict_types=1);` then (for class files) the `Ink\…` namespace and (for the bootstrap) the `ABSPATH` guard; all classes are `Ink\`-namespaced PascalCase, methods camelCase, the procedural/global surface `ink_` snake_case; the text-domain literal is `ink-core` everywhere; **no business-logic markers leak** (zero `register_post_type`, `register_taxonomy`, `register_meta`, `ink_writer_tier`, `comments_open`/`pings_open` filter, WC Membership call, REST route, `$wpdb` query, `dbDelta` table DDL — those belong to Epics 2/4/5/… and Stories 1.8/1.12); the autoloader correctly maps a sample `Ink\Kernel\Plugin` to `src/Kernel/Plugin.php`; and a structural check passes (balanced braces/quotes/PHP tags, exactly one bootstrap, one `declare(strict_types=1)` per file, valid namespace/class declarations). **`php -l` and Pest are deferred to Story 1.11** (no PHP binary / no test harness in-repo — same deferred-verification precedent as Stories 1.1–1.6). _[Source: project-context.md "declare(strict_types=1) in all ink-core PHP", "Prefix everything ink_/Ink\", escape-on-output/sanitise-on-input, "WPCS … PHPStan-clean"; architecture.md Naming/Structure patterns (lines 817–852); epics.md Story scope boundaries 1.8/1.10/1.11/1.12, Epic 2; 1-1…1-6 deferred-`php -l`/harness precedent]_

## Dev Notes

### What this story is (and is NOT)

- **IS:** the **first plugin** in the repo — the `ink-core` **bootstrap** (header + `declare(strict_types=1)` + `ABSPATH` guard + `INK_CORE_*` constants), a **PSR-4 autoloader** (`composer.json` `Ink\` → `src/` **plus** a hand-rolled `spl_autoload_register` fallback for the no-`vendor/` repo state), the **AD-1 modular-monolith `src/` tree** (Kernel + reserved module dirs each with the canonical `Module.php`/`Api.php` skeleton), a **Kernel `Plugin` bootstrap class** that wires hooks and exposes the module-registration seam, **activation/deactivation hooks** (versioned `ink_core_db_version` option stub + empty schema-registry call + `flush_rewrite_rules()` stub + PHP/WP min-version guard), and **text-domain loading** (`load_plugin_textdomain('ink-core', …)` on `init`, `languages/` reserved, no English `.mo`). All `Ink\` namespaced, `ink_` prefixed on the procedural surface, strict-typed, PHP 8.3 idioms, escaped/i18n where the scaffold itself emits a string. _[Source: epics.md#Story-1.7; architecture.md "Hand-rolled Composer/PSR-4 scaffold" (lines 682–698), AD-1 (lines 101–167), Structure patterns (lines 845–852)]_
- **IS NOT:** registering CPTs/taxonomies/user-meta (**Epic 2**); tier/Gradering, entitlement, submission, follow, engagement, challenge, sponsor, discovery logic (**Epics 4–14**); comment-disable filters (**Story 1.8 — next**); the locale-`af`/i18n-content + admin-language mechanism (**Story 1.10**); the Pest/PHPUnit + wp-env test harness and CI pipeline (**Story 1.11**); the notifications / form-letter options store (**Story 1.12**). The Kernel **declares the seam** for module bootstraps and the **empty schema registry**; later stories fill them. Reserve module dirs + facade placeholders as **extension points only** — implement **no** feature. _[Source: epics.md Stories 1.8/1.10/1.11/1.12, Epic 2; AD-9 (1.12 notifications); project-context.md three-layer]_

### Why hand-rolled-Composer-PSR-4 (autoloader rationale)

The architecture **explicitly selects** a *"Hand-rolled Composer/PSR-4 scaffold"* for `ink-core` — *"Composer autoload `Ink\` → `src/`; `declare(strict_types=1)`; module directories per AD-1, each with its own bootstrap + public `Api` facade"* (lines 682, 691–693), rejecting `wp scaffold plugin` (flat/procedural, no Composer/PSR-4/strict-types — *"Conflicts with AD-1"*, lines 675–676) and third-party boilerplates (*"largely pre-PSR-4 … net negative against AD-1"*, lines 679–680). So **Composer PSR-4 is the sanctioned mechanism** and `composer.json` declares it.

But AD-4 says *"core and third-party plugins are fetched at build, not vendored into VCS … Only `ink-core` + `ink-foundation` are committed"* (lines 339–342) — there is **no committed `vendor/`**, and CI (`composer install` → PHPStan/WPCS/Deptrac/Pest) is wired in **Story 1.11** (AD-8 lines 555–562, 720–721). In-repo today there is no PHP binary and no network install. **Therefore the scaffold ships a `src/autoload.php` bridge:** prefer Composer's `vendor/autoload.php` when present (CI/built), else register a tiny PSR-4 `spl_autoload_register` closure mapping `Ink\` → `src/`. This satisfies the architecture (Composer is the mechanism, declared in `composer.json`, optimized in CI) **and** keeps the plugin loadable in the committed repo state before 1.11 builds `vendor/`. This is the minimal correct foundation; do **not** `require` individual class files. _[Source: architecture.md lines 682, 691–693, 675–680, 339–342, 555–562, 720–721; AD-4; AD-8; epics.md#Story-1.11]_

## Review Findings

Code review (2026-06-21): adversarial three-layer review (Blind Hunter — diff-only; Edge Case Hunter — diff + project; Acceptance Auditor — diff + spec). **Failed layers: Acceptance Auditor (multiple HIGH AC violations), Edge Case Hunter (live double-`plugins_loaded` wiring activates out-of-scope features).** Blind Hunter found no fatal correctness/autoload/PSR-4/namespace bug — the scaffold's structural foundation (strict_types in all 38 PHP files, ABSPATH guard everywhere, no closing `?>`, PSR-4 closure + traversal guard + vendor-or-fallback bridge, composer.json validity, activation min-version guard, idempotent singleton, empty Schema registry, Kernel-owned enums, conflation-rule separation in Entitlement/Tiers) is correct and in-scope. The findings below are **scope-leakage** defects: later-story feature code present in (and live-wired by) 1.7 files, plus false verification claims. Triage deduplicated overlapping layer hits into single findings.

- [ ] [Review][Decision] Story 1.8 + 1.12 feature code is present and LIVE-wired by the 1.7 bootstrap — The bootstrap (`wp-content/plugins/ink-core/ink-core.php:59-65`) adds a SECOND `plugins_loaded` callback that calls `Plugin::instance()->addModule('engagement', new Engagement\Module())` and `addModule('notifications', new Notifications\Module())`. Because `boot(...)` (wired first on the same hook) dispatches `registerModules()` on `init`, both modules are in the map before `init` fires, so their `register()` runs at runtime. `Engagement\Module::register()` instantiates `Engagement\Comments` which wires `comments_open`/`pings_open` filters, `remove_post_type_support`, comment-menu/admin-bar/dashboard removal (the Story 1.8 comment-disable feature — `src/Engagement/Comments.php`). `Notifications\Module::register()` builds `TemplateStore`/`Notifier` and calls `Api::bootstrap()` (the Story 1.12 / AD-9 form-letter+`wp_mail` feature — `src/Notifications/{Module,Api,Notifier,TemplateStore,Template,MergeResolver}.php`). This violates AC-4 ("zero feature logic … no comment filters, no REST"), AC-8 (no `comments_open`/`pings_open`/`wp_mail` markers; the leak scan must fail the story), the ⚠️ guardrail "NO business logic — this is a scaffold", and the explicit scope boundaries (comment-disable = Story 1.8; notifications = Story 1.12). DECISION: either (a) revert 1.7 to a pure scaffold — delete `src/Engagement/Comments.php` + the six `src/Notifications/*` feature files, restore `Engagement`/`Notifications` to placeholder `Module.php`+`Api.php` (no-op `register()`, empty facade) per AC-3, and remove the second `plugins_loaded` `addModule` block from `ink-core.php` so the dispatcher is genuinely a no-op (as AC-4 / Completion Notes claim); OR (b) formally re-scope: fold Stories 1.8 and 1.12 into 1.7, update this story's ACs/File List/scope and the affected stories' status — but that contradicts the dependency-ordered plan and the story's own IS-NOT section. Recommended: (a).
