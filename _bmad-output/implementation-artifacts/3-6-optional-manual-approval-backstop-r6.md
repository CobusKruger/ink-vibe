# Story 3.6: Optional manual-approval backstop (R6)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a redakteur,
I want an off-by-default approval queue,
so that I can require manual approval only if abuse warrants it, without harming the default frictionless signup. (FR-3a, C8)

## Acceptance Criteria

1. **The backstop is OFF by default; when OFF, signup is exactly as frictionless as today (UJ-1) — zero added friction, zero pending state.** Given a fresh install / the default configuration, when a besoeker registers, then the account is created and immediately usable (gratis lid, Brons — the Story 3.1 `user_register` defaults) with **no "wag op goedkeuring" state, no login block, and no approval step** — identical to the current behaviour. The enabling toggle is a single `ink_`-prefixed option that **reads fail-safe OFF** (`get_option( …, false )`); an unset/garbled option value is treated as OFF. This is the hard constraint: manual approval may **never** be the launch default — it is an opt-in escalation lever only. _[Source: epics.md#Story-3.6 AC "off by default … when off, signup stays frictionless (UJ-1)"; epics.md line 143 (UJ-1: "frictionless signup; no reader/writer gate"); epics.md line 136 + sprint-change-proposal-2026-06-20.md (C8 resolved "Layered: … manual approval an optional, off-by-default backstop"); 3-4-anti-spam-decision.md (3.6 "OFF by default", the escalation lever / layer 5, composes independently with the always-on layers); project-context.md fail-safe-OFF option precedent (Notifications `TemplateStore::isEnabled`)]_

2. **When a redakteur enables it, new accounts enter a "wag op goedkeuring" state and an approval queue (admin screen) exists.** Given the toggle is ON, when a besoeker registers, then the new account is stamped with an `ink_`-prefixed **pending-approval user-meta flag** on `user_register`, and the account **cannot log in** until approved — login is blocked via a WordPress-native auth filter (`wp_authenticate_user` / `authenticate`) returning a `WP_Error` carrying an **Afrikaans-source** message (never English from `ink-core`); WordPress's credential mechanism is **used, not reimplemented**. A **redakteur** (the `editor` role) sees an admin **approval-queue screen** (the plugin's first admin UI — WP-admin chrome, no design-system work) listing pending accounts, gated on the existing `Ink\Kernel\Capabilities::MODERATE` (`ink_moderate`) capability. _[Source: epics.md#Story-3.6 AC "new accounts enter a 'wag op goedkeuring' state and an approval queue (admin screen) exists"; epics.md line 154 (state pattern "Pending approval (wag op goedkeuring) optional/off-by-default (R6)"); EXPERIENCE.md line 86 ("Rekening-goedkeuring-tou" — account-approval queue, holds new accounts in "wag op goedkeuring" until a redakteur approves/rejects), line 135 ([NEW — R6] pending state); architecture.md AD-6 (editorial actions = static custom caps granted to `editor`; `ink_moderate`); 3-4-anti-spam-decision.md (redakteur flips one ink-core setting → admin approval queue); Capabilities.php (`MODERATE='ink_moderate'`, granted to editor+admin at activation)]_

3. **A redakteur can approve (and reject) a pending account from the queue; approving clears the pending state so the lid can log in.** Given a pending account in the queue, when the redakteur **goedkeur** (approves) it, then the pending flag is cleared and the lid can log in (and is optionally notified by an Afrikaans email); when the redakteur **verwerp** (rejects) it, then the account is removed/blocked per the documented reject behaviour. Every approve/reject is a **state-changing path**: nonce + `current_user_can( MODERATE )` + sanitised input, **no raw `$_POST`/`$_GET`**. The approve/reject/pending Afrikaans verbs and the queue label are **controlled-vocabulary** — they MUST be added to `docs/afrikaans-terms.md` first, projected into the `ink-core` terminology registry (AD-10 / Story 2.0), and consumed via the registry, **never inlined as bare literals**, and **never AI-translated** (the curated-copy gate). _[Source: epics.md#Story-3.6; EXPERIENCE.md line 86 ("approves/rejects"); architecture.md AD-10 (terminology label registry — glossary-backed `ink_term()`, never inline literal), AD-6 (`ink/v1` write contract: nonce + capability + sanitise + Afrikaans errors); project-context.md "Controlled-vocabulary UI labels come from the ink-core terminology registry … never inline a glossary label as a bare literal", "No AI-generated Afrikaans", security (escape/sanitise/nonce, no raw superglobals); afrikaans-terms.md line ~283 ("add a new concept to the glossary BEFORE it appears in code or UI")]_

4. **All member-facing pending copy and transactional approval/rejection emails are Afrikaans (Gate D); the admin queue labels are `ink-core` Afrikaans-source.** Given any string this story surfaces — the login-blocked "wag op goedkeuring" message, the admin queue screen labels/buttons, and any approval/rejection email — when it renders, then it is **Afrikaans, sentence case, "jy"-voice, zero English leakage**. The redakteur works in English WP-admin chrome but every `ink-core` label/button/status string is Afrikaans-source (`ink-core` ships **no English `.mo`**). Transactional emails route through the **Story-1.12 Notifications capability** (`Notifications\Api::registerTemplate()` / `send()`), registered Afrikaans-source with the **send toggle OFF by default** until human-authored body copy lands (`[WAG OP MENSLIKE KOPIE]` placeholder — do not AI-translate). Where the precise pending/approval copy is not yet authored, flag `[NEEDS HUMAN AFRIKAANS]` in `ui-copy-translations.md` (mirroring the line ~544 precedent, incl. the "moenie KI-vertaal nie" caveat) — do not invent it. _[Source: project-context.md Gate D (front end + transactional emails entirely Afrikaans; admin-language split — `ink-core` admin = Afrikaans source, no English `.mo`; sentence case; no AI Afrikaans); architecture.md AD-9 (Notifications composes/dispatches Afrikaans transactional email, toggle-gated; leak scan covers the options store); EXPERIENCE.md line 68/77 (redakteur: English admin chrome, Afrikaans ink-core labels); 1-12-form-letter-notification-capability-foundation.md; ui-copy-translations.md line ~544 ([NEEDS HUMAN AFRIKAANS] precedent + R7 "moenie KI-vertaal nie")]_

## Dev Notes

### What this story is (and is NOT)

- **IS:** an **off-by-default** registration approval **backstop** in the `ink-core` `Ink\Accounts` module — (1) a single fail-safe-OFF toggle; (2) a WP-native pending **"wag op goedkeuring"** account state stamped on `user_register` when ON; (3) a login gate (WP-native auth filter → Afrikaans `WP_Error`) while pending; (4) the plugin's first **admin approval-queue screen** (WP-admin chrome, redakteur, `ink_moderate`) to **goedkeur** / **verwerp** pending accounts; (5) Afrikaans pending copy + approval/rejection emails via the Story-1.12 Notifications capability (toggle-OFF until human copy); (6) authored AND passing Pest tests. _[epics.md#Story-3.6; 3-4-anti-spam-decision.md; architecture.md AD-1/AD-6/AD-9/AD-10]_
- **IS NOT:** social login (**Story 3.5** — separate, hooks-only OAuth); the always-on anti-spam baseline — **Cloudflare Turnstile, email double-opt-in, honeypot/timing** (the Story-3.4 spike's layers 1–4; not a 3.6 deliverable); **edge rate-limiting, IP reputation, Patchstack abuse monitoring, Turnstile tuning, blocked-attempt analytics** (**Story 18.10** — which *hardens around* the pending state that 3.6 builds); any change to the OFF default; any front-end / design-system / `theme.json` / Lovable work (the queue is WP-admin chrome); any `Ink\Entitlement`/`Ink\Tiers`/lidmaatskap/Gradering coupling; reimplemented WordPress auth.

### ⭐ The decided approach (key deliverable) — a thin `ink-core` seam over WP-native account status

The Story-3.4 build decision **decides** this — do not re-litigate it:

> The approval queue is **layer 5** of a 5-layer defense-in-depth — the **escalation lever only**, not always-on. The redakteur flips **one `ink-core` setting**; when ON, new accounts enter **"wag op goedkeuring"** and surface in an admin queue. It composes **independently** with the always-on layers ("Turnstile stops bots at the edge; the approval queue is for *human* abuse the automated layer can't judge"). R6 stays a **vetted-plugin / edge seam via hooks, NOT `ink-core` anti-spam business logic** (architecture.md line 650–652) — but the toggle + pending-state + queue are the legitimate `ink-core` Accounts-module surface (the R6 seam lands in Accounts/Epic 3, architecture.md line 651).

So the shape is unambiguous:
- **WordPress** owns the auth *mechanism* and account record — the pending state rides a WP-native user-meta flag + a login *filter* (`wp_authenticate_user`/`authenticate`), never a reimplemented credential path.

Prefer the thin `ink-core` seam; if the dev finds BuddyPress's signup-approval is already active and a better fit, **hook it, do not reimplement** — and document the choice. _[project-context.md "Hook, don't edit"; "don't build a feature a kept plugin already covers"]_
