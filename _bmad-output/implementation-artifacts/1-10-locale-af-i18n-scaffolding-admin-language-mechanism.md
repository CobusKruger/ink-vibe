# Story 1.10: Locale af + i18n scaffolding & admin-language mechanism

Status: in-progress

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Afrikaans-first product,
I want the locale (`af`), i18n scaffolding, and the admin-language mechanism established in the Foundation phase — by building out the i18n seam Story 1.7 reserved: a correct/complete `ink-core` text-domain load, theme (`ink-foundation`) text-domain loading for its custom front-end strings, a per-user admin-language forcing mechanism that pins editor/administrator accounts to English admin while leaving the site/front-end locale `af` untouched, and the standing no-English-`.mo` policy that makes `ink-core`'s Afrikaans-source admin strings fall through to Afrikaans under that forced-English admin locale,
so that Afrikaans-first is built-in rather than retrofitted (Principle 3 / §12): every custom front-end string renders Afrikaans via gettext with the correct text domain, staff get a findable English admin chrome without leaking English to visitors, and `ink-core`'s own admin labels stay Afrikaans even under that English admin locale.

## Acceptance Criteria

1. **Site locale `af` → custom front-end strings are internationalised via gettext with the correct text domain and render Afrikaans (sentence-case headings).** Given the site locale is `af`, when any custom string is output on the front end, then it is wrapped in a proper gettext call (`__()`/`_e()`/`esc_html__()`/`_n()`/etc.) with the **correct text domain** — `ink-foundation` for theme-emitted strings, `ink-core` for plugin-emitted strings — and renders Afrikaans, with sentence-case headings (Gate D). The text-domain load paths that make this resolve are in place: `ink-core` loads its `ink-core` domain from its `/languages` dir (the Story 1.7 `Kernel\I18n` stub built out and confirmed wired on `init`), and the theme loads its `ink-foundation` domain via `load_theme_textdomain()` so its own `__( …, 'ink-foundation' )` strings (e.g. the 1.5 pattern-category labels in `functions.php`) resolve. The site locale `af` itself is **carried forward on the brownfield DB / `wp_options`** (`WPLANG`/`locale` option), not hard-switched in code that fights WordPress — this story documents that mechanism and ensures the i18n wrapping + domain loading are correct so `af` resolves. _[Source: epics.md#Story-1.10 AC-1 "Given site locale `af`, When any custom string is output on the front end, Then it is internationalised via gettext with the correct text domain and renders Afrikaans (sentence-case headings)"; project-context.md "i18n on every user-facing string with the correct text domain (`ink-core`/`ink-foundation`)", "Heading casing: sentence case", "carry forward only deliberate values (site URL/name, `af` locale)"; docs/specs/ink-consolidated-spec.md §14 row 11 + line 353 "Locale mechanism: site locale `af` … All custom strings use proper i18n functions" + line 345 "carry forward only deliberate values (site URL/name, `af` locale)"; architecture.md line 706 "1.10 (theme half of i18n scaffolding)", lines 129–130 Kernel "i18n loader"; 1-7 `src/Kernel/I18n.php` stub]_

## Dev Notes

### What this story is (and is NOT)

- **IS:** the i18n **foundation** in `ink-core` (Kernel) + the theme, building out the seam Story 1.7 explicitly reserved. Four pieces: (1) a correct/complete `ink-core` text-domain load in `Kernel\I18n` (right domain, `/languages` Domain Path, `init` hook); (2) the **admin-language forcing mechanism** — an `ink_`-prefixed `get_user_locale` filter (Kernel i18n concern) that returns `en_US` for editor/administrator users **only in admin context**, leaving the front-end/site locale `af` untouched (§14.14); (3) the **no-English-`.mo` policy** for `ink-core` (Afrikaans authored as gettext source; ship no English `.mo`; gettext falls through to the Afrikaans source under the forced-English admin locale — §14.15), enforced via the `languages/` dir holding only `.gitkeep` (+ optional Afrikaans artifact) and documented as the standing rule; (4) theme-side `load_theme_textdomain('ink-foundation', …)` so the theme's own Afrikaans-source strings resolve. Wired through the **existing 1.7 Kernel seam** (`Kernel\Plugin` wires `Kernel\I18n`), not a parallel mechanism. _[Source: epics.md#Story-1.10; architecture.md Kernel "i18n loader" (129–130), line 413; 1-7 `Kernel\I18n`/`Kernel\Plugin`; §14.14/§14.15]_
- **IS NOT:** hard-switching the site locale in code (locale `af` rides the brownfield DB / `wp_options`, set in migration 16.11 — this story documents the mechanism, it does not fight WP by force-setting the locale at runtime); registering CPTs/taxonomies or their labels (Epic 2 — though those labels will follow this Afrikaans-source rule); building any `ink-core` admin screen (tier promotion, sponsor scheduling, challenge/winner admin, reports ship with their features in later epics); authoring UI copy at volume or running the English-leak scan (Epic 17 — execution/QA; 17.4 owns the standing leak gate); creating `wp-content/languages/` or committing any 3rd-party `.mo` (Epic 17 / translation workflow); standing up the Pest runner / CI (Story 1.11). _[Source: epics.md Stories 1.10/1.11/2.1/2.2/17.4, Epic 16.11; project-context.md translation workflow; docs/specs §14]_

### Why the Kernel (placement decision)

The architecture lists **"i18n loader"** as a **Kernel** concern (lines 129–130: *"Kernel (enums · bootstrap · PSR-4 autoload · dbDelta schema registry · capabilities · i18n loader)"*), and the admin-language split is described as cross-cutting (line 413). Story 1.7 already placed text-domain loading in `Kernel\I18n` and wired it from `Kernel\Plugin`. The admin-language forcing is the same cross-cutting i18n concern (it governs *all* admin chrome locale resolution, not one feature's domain), so it belongs alongside `I18n::load()` in the Kernel — either as a method on `Kernel\I18n` or a small sibling `Kernel\AdminLanguage` collaborator (mirroring how 1.8 kept `Engagement\Module` thin by delegating to a `Comments` collaborator). A feature-module placement was rejected — there is no feature domain that owns "what language is the admin in"; it is a platform-wide policy. The theme `load_theme_textdomain` call is the one piece that lives in the theme, and correctly so: it loads the *presentation* layer's own strings (the theme's `ink-foundation`-domain labels), which is presentation infra, not business logic — the three-layer rule forbids business logic in the theme, not the theme loading its own text domain. _[Source: architecture.md lines 129–130, 413; 1-7 `Kernel\I18n`/`Kernel\Plugin`; 1-8 collaborator house style; project-context.md three-layer]_

### How the 1.7 Kernel i18n seam works (build it out, don't reinvent)

From `src/Kernel/Plugin.php` (1.7): `boot()` is an idempotent singleton wired on `plugins_loaded`; `run()` wires `add_action('init', I18n::load( ... ))` (first-class callable) and `add_action('init', $this->registerModules( ... ))`. From `src/Kernel/I18n.php` (1.7): `final class I18n` with `const TEXT_DOMAIN = 'ink-core'` and `static load(): void` calling `load_plugin_textdomain(...)`; its docblock reserves the 1.10 build-out. **This story's wiring:** complete/confirm `I18n::load()`, add the admin-language filter callback (a static method on `I18n` or a `Kernel\AdminLanguage` class), and register `add_filter('get_user_locale', …)` from `Kernel\Plugin::run()` next to the existing `init` wiring (the Kernel is the single place cross-cutting concerns are wired). Do **not** add a parallel `add_filter` in `ink-core.php` — that bootstrap only boots the Kernel and registers feature modules; cross-cutting i18n is the Kernel's job. _[Source: 1-7 `src/Kernel/Plugin.php` `run()`, `src/Kernel/I18n.php`; 1-8 "use the Kernel seam, not a parallel mechanism"; architecture.md AD-1]_

### ⭐ The exact mechanism (key deliverable)

**The admin-language forcing — filter `get_user_locale`:**

| Aspect | Decision |
|---|---|
| Hook | `add_filter( 'get_user_locale', $callback, 10, 2 )` — wired from `Kernel\Plugin::run()` |
| Callback signature | `( string $locale, int|\WP_User $user_id ): string` (matches WP's `get_user_locale` filter args) |
| Guard 1 (context) | if `! is_admin()` → `return $locale;` (front end is untouched — it uses `get_locale()`, not the user locale) |
| Guard 2 (role/cap) | resolve `WP_User`; if `user_can( $user, 'edit_others_posts' )` (editor/administrator-defining cap) or explicit `editor`/`administrator` role → `return 'en_US';` else `return $locale;` |
| What it returns | `'en_US'` for staff in admin; the incoming `$locale` (i.e. `af`) for everyone else / front end |
| Why this filter | read-time, idempotent, no DB mutation; `load_*_textdomain`/admin locale resolution consult it; cannot leak to the front end (front-end output uses site `get_locale()`). §14.14. |
| Anti-pattern avoided | **NOT** `update_user_meta($id,'locale','en_US')` on read — mutation-on-read fights a staff member who set their own preference and is not idempotent. The read-time filter is the documented robust approach. |

- **Front-end-stays-`af` guarantee:** WordPress resolves front-end output locale via `get_locale()`/`determine_locale()` (the **site** locale = `af`), which does **not** consult `get_user_locale`. The `is_admin()` guard is belt-and-braces so the filter is a strict no-op anywhere but wp-admin. This is the load-bearing half of AC-2.
- **`en_US` choice:** the spec/decision says "English"; `en_US` is WP's canonical built-in English locale (no language pack needed). If the org later prefers `en_GB`, it is a one-constant change.

**The text-domain loads:**

| Domain | Loader | Hook | Where |
|---|---|---|---|
| `ink-core` | `load_plugin_textdomain('ink-core', false, dirname(plugin_basename(INK_CORE_FILE)).'/languages')` | `init` | `Kernel\I18n::load()` (built out from 1.7) |
| `ink-foundation` | `load_theme_textdomain('ink-foundation', get_template_directory().'/languages')` | `init` | `ink-foundation/functions.php` (new) |

**The no-English-`.mo` policy (§14.15) — how it works:** gettext, under locale `en_US`, looks for `ink-core-en_US.mo`. Because `ink-core` ships **none**, gettext finds no translation and returns the **untranslated source literal** — which is the Afrikaans string the developer wrote (`__( 'Kennisgewing', 'ink-core' )` → "Kennisgewing"). So a staff member on forced-English admin sees English WP-core/3rd-party chrome (those *do* have `en_US` packs) but Afrikaans `ink-core` labels. Enforcement = the `languages/.gitkeep` prohibition + this rule + the 17.4 leak scan. There is nothing to "do" at runtime — the policy is the *absence* of an English `.mo`.

### The locale-`af` mechanism (document, don't fight WP)

Site locale `af` is a `wp_options` value (`WPLANG`/`locale`) that **rides the brownfield DB clone** — project-context.md "carry forward only deliberate values (site URL/name, `af` locale)"; the carry-forward itself is migration Story **16.11** (selective options carry-forward). This story does **not** set or hard-switch the locale in PHP (that would fight WP and break the Site Editor / language dropdown). What this story owns is making the i18n *plumbing* correct so that **when** the locale is `af`, custom strings resolve Afrikaans: proper gettext wrapping with the right domain (`ink-core`/`ink-foundation`) and the domain loaders on `init`. The Afrikaans-source authoring means front-end `ink-core`/theme strings render Afrikaans directly (their source literal is Afrikaans); 3rd-party front-end strings pull `af` language packs / committed `.mo` (Epic 17). _[Source: project-context.md "carry forward only deliberate values (site URL/name, `af` locale)"; docs/specs §14 line 345/353; epics.md 16.11]_

### ⚠️ Guardrails (prevent disasters)

- **NEVER ship an English `ink-core` `.mo`/`.po`.** Afrikaans is the gettext SOURCE; shipping `ink-core-en_US.mo` would *override* the Afrikaans source under the forced-English admin locale and defeat §14.15. `languages/` holds only `.gitkeep` (+ optional Afrikaans). _[§14.15; 1-7 `.gitkeep`]_
- **NEVER let the admin-language filter touch the front end.** Guard on `is_admin()` and return `$locale` unchanged when false. The front end must stay `af` for every user, staff included. _[§14.14; AC-2]_
- **NEVER mutate user-meta on read.** Use the read-time `get_user_locale` filter, not `update_user_meta`. Idempotent, non-destructive, respects a staff member's own setting outside the forced scope. _[Task 2 rationale]_
- **NEVER AI-translate or invent Afrikaans.** Any string added comes from `docs/afrikaans-terms.md` / `docs/ui-copy-translations.md`; sentence case (Gate D). _[project-context.md "No AI-generated Afrikaans", "afrikaans-terms.md is the glossary source of truth", "Heading casing: sentence case"]_
- **NEVER hard-switch the locale in code.** `af` rides the DB (16.11). This story is plumbing + the admin-language filter, not a runtime locale setter. _[project-context.md options carry-forward]_
- **Build out the Kernel seam, not a parallel mechanism.** Wire the i18n concerns from `Kernel\Plugin`; do not add `add_filter` directly in `ink-core.php`. _[1-7; 1-8 discipline; AD-1]_
- **`declare(strict_types=1);` + `ABSPATH` guard in every new/edited `ink-core` `.php`.** `Ink\Kernel`-namespaced PascalCase classes, camelCase methods, `ink_` snake_case on any global/hook surface, no raw superglobals. Theme file keeps its presentation-only header. _[project-context.md]_
- **Theme gets ONLY the textdomain loader** — no business logic, no locale logic, no role logic in the theme. _[project-context.md three-layer]_
