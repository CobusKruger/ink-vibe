# Story 4.3: Access enforcement (entitlement gate)

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As the system,
I want to grant submission entitlement iff a lid has an active WooCommerce Membership,
so that entitlement tracks lidmaatskap and nothing else. (FR-6, FR-13, FR-19)

## Acceptance Criteria

1. **Active lidmaatskap ‚áí the betaalde lid may plaas; the gate evaluates the membership END DATE in SAST, not the cron-flipped status flag.** Given a lid whose WooCommerce Membership end date is in the future (or whose end date's calendar day has not yet passed end-of-day SAST), when entitlement is evaluated, then `Ink\Entitlement\Api::can_submit( $user )` returns `true`. The authority is the membership **end date compared to "now" in SAST** ‚Äî NOT WooCommerce's stored `expired`/`active` status string, which is flipped by an Action Scheduler / wp-cron job that may lag the true expiry instant. A lidmaatskap is valid **through end of day SAST** on its expiry date (`23:59:59 SAST = 21:59:59 UTC`). This is deterministic and cron-independent: no scheduled job needs to have run for the gate to return the correct answer. _[Source: architecture.md AD-2 ("`can_submit()` evaluates against the membership end date compared to current time, not the cron-flipped status flag ‚Ä¶ it can honour 'valid through end of day SAST' exactly, with no lag window"), AD-2 Granularity ("the gate treats the end date's calendar day as valid through 23:59:59 SAST = 21:59:59 UTC; times are stored in UTC and compared in SAST"); epics.md#Story-4.3 AC ("Given an active lidmaatskap ‚Ä¶ the betaalde lid may plaas"); epics.md FR-6 (line 43, P0); project-context.md THE conflation rule ("subscription status (active WooCommerce Membership) controls submission entitlement")]_

2. **The end-of-day SAST boundary is computed by a single, reusable SAST helper ‚Äî the same helper AD-3 challenge open-window checks consume.** Given the "valid through end of day SAST" policy, when the boundary is computed, then it is computed once in a single source-of-truth SAST helper (a static method, reusable by both this gate and the AD-3 challenge deadline / entry-freeze), in the `Africa/Johannesburg` timezone (UTC+2, no DST) ‚Äî NOT in UTC and NOT in the server's local timezone. Just-before-`23:59:59 SAST` on the expiry day is still entitled (`true`); just-after the boundary is denied (`false`). One implementation of "end of day SAST", reused ‚Äî never two. _[Source: architecture.md AD-2 ("Single SAST helper: the same SAST boundary helper serves both this gate and the challenge deadline / entry-freeze (FR-47, inclusive 23:59:59 SAST). One source of truth for 'end of day SAST,' reused ‚Äî not two implementations"), AD-3 decision 3 ("Open-window check reuses the single SAST boundary helper (AD-2)"); project-context.md ("Model fixed value sets ‚Ä¶ never duplicate these literals"); SAST = Africa/Johannesburg, UTC+2, no daylight saving]_

3. **Expiry / suspension auto-revokes entitlement ‚Äî emergently, with NO write that deletes the account, changes `ink_writer_tier`, or unpublishes bydraes.** Given a lid whose membership end date has passed (yesterday or earlier), when entitlement is evaluated, then `can_submit()` returns `false` **regardless of whether WooCommerce's stored status still says `active`** (the lag case). "Auto-revoke" is **emergent, not a stored flag or a scheduled action**: because the gate reads the end date live, a lapsed membership simply makes `can_submit()` return `false` ‚Äî there is no separate revoke routine, and crucially the denial path performs **NO write** of any kind. The account is never deleted (`wp_delete_user`), `ink_writer_tier` is never touched (`update_user_meta( ‚Ä¶, 'ink_writer_tier', ‚Ä¶ )`), and no published bydrae is unpublished or status-changed (`wp_update_post` / `wp_trash_post`). The expired betaalde lid silently reverts to a gratis lid (no submission entitlement) while account, Gradering, and published work persist. _[Source: epics.md#Story-4.3 AC ("on expiry/suspension entitlement auto-revokes (reverts to gratis lid)" AND "revoking entitlement does not delete the account, change `ink_writer_tier`, or unpublish bydraes"); architecture.md AD-2 ("Status flag is for passive reads only ‚Ä¶ It must not be the authority for the publish gate"); project-context.md edge case ("Expired/suspended membership ‚Üí submission denied, but the account, its tier, and published work persist")]_

4. **Fail-safe deny: no membership / WooCommerce Memberships inactive / null or logged-out user ‚áí `can_submit()` returns `false`.** Given any of: a `null` user, a logged-out request, a user with no WooCommerce Membership at all, or WooCommerce Memberships being inactive/absent (its API functions undefined), when entitlement is evaluated, then `can_submit()` returns `false` ‚Äî never a fatal, never an over-permissive `true`. The membership is read through WooCommerce Memberships' own API (`wc_memberships_get_user_active_memberships()` / `wc_memberships_is_user_active_member()` and the membership object's `get_end_date()` getter), behind `function_exists` / availability guards so the seam degrades gracefully when the platform plugin is absent (mocked in tests; do NOT reimplement membership storage). The "active lidmaatskap" is one tied to a Story-4.1 lidmaatskap plan/product. _[Source: architecture.md AD-2 ("Failure direction is benign ‚Ä¶ never over-restrictive against a valid member" ‚Äî but a non-member must be denied), AD-5 ("membership in the WC store"), project-context.md ("do not reimplement" platform plugins; WooCommerce Memberships is a commodity capability); MembershipPlans::isWooCommerceAvailable() precedent (a protected, test-overridable availability seam ‚Äî Brain Monkey-defined symbols persist within a process)]_

### ‚≠ê The decided approach (key deliverable) ‚Äî end-date-in-SAST evaluation, cron-independent

AD-2 **decides** this ‚Äî do not re-litigate it:

> The Entitlement module's `can_submit()` evaluates against the membership **end date compared to current time**, **not** the cron-flipped status flag. This is deterministic and cron-independent ‚Äî it can honour "valid through end of day SAST" exactly, with no lag window, because no scheduled job needs to have run for the gate to return the correct answer. ‚Ä¶ "end of day SAST" = the end date's calendar day valid through **23:59:59 SAST = 21:59:59 UTC**. ‚Ä¶ The same SAST boundary helper serves both this gate and the challenge deadline / entry-freeze. One source of truth.

So the shape is unambiguous:
- **WooCommerce Memberships** owns the membership record + status; INK **reads** it via the platform API and **never** writes/flips the status (use, don't reimplement).
- **`Ink\Kernel\Sast`** owns the ONE "end of day SAST" boundary computation (UTC+2, no DST), reused by AD-2 here and AD-3 challenges.
- **`Ink\Entitlement\SubmissionGate`** (facaded by `Api::can_submit()`) owns the gate logic ‚Äî read the INK-lidmaatskap membership end date, compare to "now" in SAST, return bool. Pure read; auto-revoke is emergent.
